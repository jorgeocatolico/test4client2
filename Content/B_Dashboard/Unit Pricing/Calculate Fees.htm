Calculate Fees

The calculation of fee provisions for an entity is a key function of the pricing process and the pControl unit pricing product provides a very flexible and highly configurable fee calculation framework. An entity can have as many fees as required calculated for it, and the following sections explain how to configure fee entities in pControl and how to associate them to the entity so that fee provisions are calculated during a pricing process.

Configuring Fee Entities

A ‘fee’ entity is created in the process tree and the attributes held on a fee entity define what type of fee it is and what calculations will take place when it is calculated for an entity.

Note

See Processing Volume I User Guide and System Administration User Guide for more information on the creation and maintenance of entities in pControl.

The key attributes used for the configuration of fee entities are:

  • fee type
  • fee method
  • base value.

The frequency that the fee is applied, and the rates to apply are held on the entity itself unless the fee represents a sliding scale.

 Fee Type

The fee type defines whether the fee being calculated is a fee (i.e. admin, custody, and management) or a performance fee.

Fee Type

Description

Fee

Selected for all fees that are not performance fees (i.e. admin, management, custody).

Performance Fee

Selected where the fee being calculated is a performance fee. This specifies that a performance fee is to be calculated which means that the additional attributes under the performance fee tab are used

Note: For further details on performance fee calculations see section 4.8.1.5 [LC035] Calculate Performance Fees .

 

 Fee Method

The fee method defines how a fee will be calculated. The fee methods available include:

Fee Method

Description

Percentage

The fee is calculated as a percentage of the base value. The fee percentage is held on the entity along with the frequency that it is applied.

The calculation is as follows:

(bp amount from the ‘Rate(bp)’ field * Base Value * (number of days since last price / days in the frequency))

For example, a bp rate of 120 applied annually on the Current Day GAV where the number of days since the last price is 3, the number of days in the year is 365 and the Current Day GAV is currently 15,000,000 would result in 15,000,000 * 0.012 * (3/365) - with the resulting amount being the fee to provide for on the processing day

Flat Fee

A flat fee amount is used to calculate the fee using the frequency of the fee that is held on the entity.

The calculation is as follows:

(Flat Fee $ amount from the ‘Flat Fee $’ field * (number of days since last price / days in the frequency))

For example, a flat fee of $1,100 applied annually where the number of days since the last price is 1 and the number of days in the year is 365 would result in 1000 * (1/365) - with the resulting amount being the fee to provide for on the processing day

Flat Fee and Percentage

A flat fee is used to calculate the fee plus a percentage of the base value specified – a combination of flat fee and percentage methods.

In this case, the flat fee is calculated and then the relevant percent based amount and both are summed to provide the provision for the day to accrue.

Sliding Scale Absolute

Sliding scale fee rates are used to determine the percentage rate to apply to the base value, i.e., whichever scale band the base values falls into, that rate is used along with the frequency held on the entity.

The calculation would be as follows:

(bp amount from the appropriate band on the sliding scale table on the pricing entity * base value * (number of days since last price / days in the frequency))

For example, if the current day GAV was $53,000 and using the sliding scale table below with the number of days since the last price being 2 and the number of days in the year being 365, the fee would be calculated as; 53,000 * 0.0030 * (2/365) with the resulting amount being the fee to provide for on the processing day.

 

Alternatively, there is an attribute ‘Rate represents flat monetary amount’ which if selected will mean that the ‘Rate’ column represents a ‘flat monetary amount’ rather than a basis point figure. As a comparison, if the current day GAV was $53,000 and using the sliding scale table below with the number of days since the last price being 2 and the number of days in the year is 365, this would result in the fee being calculated as 53,000 * 2,300 * (2/365) with the resulting amount being the fee to provide for on the processing day.

Note: Sliding scale fee details are defined on a ‘Fee Scale’ entity which is attached to the entity via the ‘Fee Details’ attribute table.

Sliding Scale Cumulative

The sliding scale fee rates held on the pricing entity are used with each band’s rate applied against a portion of the base value up to the band in which the base value sits

The calculation is as follows:

(bp amount from each band on the sliding scale table on the pricing entity * max amount-min amount) until base value < max amount and then (bp amount from band * base amount – min amount) * (number of days since last price / days in the frequency)

 

For example, if the GAV was 60,000 and using the following sliding scale table:

With the number of days since the last price being 2 and the number of days in the year is 365, this would result in ((20 * 50,000) + (10000 * 30)) * (2/365) with the resulting amount being the fee to provide for on the processing day.

Flat Fee and Sliding Scale Absolute

A flat fee is used to calculate the fee plus a percentage of the base value specified using the sliding scale absolute method and the sliding scale specified – a combination of flat fee and sliding scale absolute methods.

Flat Fee and Sliding Scale Cumulative

A flat fee is used to calculate the fee plus a percentage of the base value specified using the sliding scale cumulative method and the sliding scale specified – a combination of flat fee and sliding scale cumulative methods.

Return Value From Function

The business rule held in the ‘Rule to be used to generate fee value’ attribute will be used to generate a fee value. This provides flexibility in creating new fee methods to meet specific client requirements.

 

 Base Value

The base value defines the value that will be used in the calculation of a fee. The base value alternatives include:

Base Value

Description

Average GAV

The calculated average GAV of the entity since the last lock-in period will be used as the base value

Note: this option is only valid when the fee type is ‘Performance Fees’. Performance fees are configured using attributes on a separate tab of the ‘Fee Type’ entity.

Cashflow Since Last Pricing Point

The value of any cashflows received since the prior pricing point or processing day (as retrieved from the unitised cashflow worktable) will be used as the base value.

Current Day GAV

The Gross Asset Value calculated for the entity by the pricing process will be used as the base value.

Current Day GAV – Bid / Offer

The Gross Asset Value calculated for the entity by the pricing process when load bid/offer validations has been selected will be used as the base value.

Current Day GAV – Bid / Offer (Clean Market Value)

The Gross Asset Value calculated for the entity by the pricing process when load bid/offer validations and clean market value (for transaction cost and price component adjustments) have been selected will be used as the base value.

Current Day GAV less Prior Fees

GAV minus ‘Total Accrued Fee Provision’ before today (i.e. sum of all fees accruals included in today’s total fee provision).

Current Day GAV less Prior Tax

The Gross Asset Value calculated for the entity by the pricing process minus the tax provision from the previous pricing point will be used as the base value.

Individual Investment Holding Value

The value of the individual investment holding (for the investment entity as specified on the pricing entity) will be used as the base value.

Pre-Performance NAV

The calculated pre-performance NAV value will be used as the base value.

Note: this option is only valid when the fee type is ‘Performance Fees’. Performance fees are configured using attributes on a separate tab of the ‘Fee Type’ entity.

Prior Period GAV

The Gross Asset Value from the prior pricing point or processing day for the entity will be used as the base value

Prior Period GAV including cashflows since

The Gross Asset Value from the prior pricing point or processing day for the entity plus any cashflows received since that point (as retrieved from the unitised cashflow worktable) will be used as the base value

Prior Period

The Net Asset Value from the prior pricing point or processing day for the entity will be used as the base value

Prior Period incl. cashflows since

The Net Asset Value from the prior pricing point or processing day for the entity plus any cashflows received since that point (Cashflows as loaded in function [DL100] from the U109: Cashflows unitised worktable, pControl or both) will be used as the base value

Prior Period Pre-performance NAV including cashflows since

The calculated prior period pre-performance NAV including cashflows since value will be used as the base value

Note: This option is only valid when the fee method is set to ‘Performance Fee’.

Return value from Function

The business rule held in the ‘Rule to be used to generate base value’ attribute will be used to generate a value to be used as the base value. This provides flexibility in creating new base values for fees easily to meet specific client requirements.

Total Investment Holding Value

The total of the investment holdings calculated for the entity by the pricing process will be used as the base value.

Capital Value Pre-Expense

The pre-expense NAV minus the pre-expense income will be used as the base value.

Note: The pre-expense NAV = GAV minus opening liabilities balances.

Income / Expense Account Movement

The movement in account balance between current and last pricing day will be used as the base value.

Note: This base value is typically used in calculating fees for NAV Oversight entities. See NAV Oversight User Guide for further information.

 

 Additional attributes used in ‘Fee Type’ entity Configuration

Attribute

Description

Transactions Types

Transaction types on the fee entity will be used to accrue the fee that has been calculated along with any sales tax and sales tax refund that might have been calculated in addition, based on the attributes held against the entity. It is the transaction types which define the pControl internal accounts that will have their balances updated when the fee is accrued.

Non-Annualised?

If this attribute is selected, the fee will be calculated and not annualised, i.e., it will ignore the frequency of the fee set against the entity and calculate the fee provision without any ‘annualisation’. This is the equivalent of setting the fee to have a daily frequency at all times.

Fee reversal journal required for next valuation point

If selected, once the fee journal is created, a reversal journal entry will also be created reversing the current accrual for the next valuation point.

Round fee amount before accruing

If selected, fee values will be rounded before multiplying by the number of days to accrue. If not selected, the unrounded amount will be used when multiplying by the number of days to accrue.

Notes

Free format text that allows a business description to be added to the Fee Type.
