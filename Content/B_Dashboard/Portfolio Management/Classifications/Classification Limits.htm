Classification Limits

Overnight market movements and allocating cash can change the market value of an asset and therefore the market value of the sector or classification grouping. If the weight of an asset sector within a plan moves away from a 'Target', the Portfolio Manager may want to change the value of the asset, and therefore a sector by investing or divesting in that asset.

The target weights of an asset or sector within a classification and its allowable limits, i.e. how far the asset or sector value can deviate from a target value, can be configured in processing mode and web mode; with outputs displayed in the Portfolio Management classification panel in the web view.

Web Mode

It is possible to directly input the parameter values while in web mode, using the Classification Parameters view. The steps for configuring Classifications and Parameters in web mode are described in Portfolio Management Configuration.

Processing Mode

The examples provided below describes how classification limits are used and how they are configured in processing mode.

Tolerances and bounds

If the value of an asset or sector moves outside of the limits, the order may not be able to be approved, depending on whether a 'hard' or 'soft' breach is detected. There are two limits defined within pControl, 'tolerances' and 'bounds'.

  • Tolerances are the percentage movements away from a target where the portfolio is considered outside tolerance.
  • Bounds are absolute upper and lower limit where the portfolio is considered out of bounds.
Example

Percentage values could be calculated as a percentage of the plan's total value, e.g, the Target Value of USD investments is 30% of the total plan; or as a percentage of a parent classification, e.g., the Target Value of USD investments is 30% of all foreign currency classifications; or any other value / classification could be used. The value to be used is defined in the denominator field of the U496: Classification Allocation Parameters worktable.

Recording values in the U496: Classification Allocation Parameters worktable

  • Target values are entered into the U496: Classification Allocation Parameters worktable as an absolute value; and the CA_LOAD_CLASSFCATION_ALPS process is run in pControl.
  • Tolerance values are entered into the U496: Classification Allocation Parameters as positive values relative to the target, i.e. the difference between the lower tolerance and target; and the upper tolerance and target.
  • Bound values are entered into the U496: Classification Allocation Parameters worktable as absolute values.

 Classification basis

Classification basis allows pControl to choose the value to include in the classification test. This can be one of three values which can be selected in pControl as shown below.

  • Total Value
  • Confirmed Value
  • Confirmed Value (Trade)

Total Value

This is the default value that is used in classification tests. The value tested is the sum of confirmed investment value plus:

  • The value of any in-flight trades; plus
  • The value of any proposed transactions.

Confirmed Value

This value represents the confirmed investment value. Confirmed value in pControl means the sum of all trades where trade date is less than or equal to effective date and the order is confirmed.

Confirmed Value (Trade)

Confirmed Value on a trade date basis, is a similar method to the confirmed value, but assumes that all trades are confirmed at their confirmation date i.e. the sum of all trades where trade date is less than or equal to effective date and order is confirmed or unconfirmed.

 

 

 Soft / hard breaches

Whether a breach is a hard or soft, is determined by the 'Compliance Checks' attribute within the Cash Allocation tab on entities within the processing mode.

  • Soft breaches can be overridden, to allow an order to be approved.
  • Hard breaches cannot be overridden. Instead, the order's value needs to be changed to move the entity's value and the classification's value back inside the limits.

Up to 5 classification checks can be configured in pControl. For each check configured, both the tolerance check and bound check needs to be defined as either a soft or hard breach using the drop-down menus below.
