Expected Pricing Methodology

The NAV price comprises several categories of assets and expenses. Each category of the NAV can be calculated differently, using different Expected Return Methodologies. These methodologies are then applied to their respective categories using the Expected Pricing Methodology entity.

For example, the NAV for an Index Tracking Fund might comprise four categories:

  • Investment Assets (AST)
  • Fixed Interest Accrual Assets (FIAI)
  • Fees (FEE)
  • Transactions (TXN).

Each of these categories are calculated differently, and then summed to determine the overall expected NAV. One Expected Pricing Methodology entity could be configured with these four category calculations, so as to be used for expected pricing for all Index Tracking funds.

To configure the Expected Pricing Methodology entity, as shown below, select a category and Expected Return Methodologies from each of the drop down fields.

Note

pControl will process each category in the order listed in the table below, however this will not alter the outcome of the overall NAV, i.e. each category will be calculated separately, and then summed to produce a total NAV.

Income Tab

Income for non working days such as weekends and public holidays, when expected pricing is not run needs to be estimated, and this is done via the use of a ‘stale’ income value, i.e. either Friday’s value is re-used as the ‘stale value’ on Saturday and Sundays, or Monday’s value could be re-used as the ‘stale value’ on Saturday and Sundays.

 

Income Stale Attribute Description

Policy for accruing non work days

A drop down list of options for accruing income over non working days such as weekends and public holidays, when expected pricing would not be run.

Drop down values include:

  • After non-working period

On the first working day after non working periods, the Expected Pricing process will estimate the income for Monday, and apply this value for both Saturday and Sunday as the ‘stale value’. Therefore, when Expected Pricing is run on Monday, the income amount will represent Saturday, Sunday and Monday inclusive.

  • Before non-working period

On the last working day before a non working period, the Expected Pricing process will calculate the income for the current day and apply it to the non working period as a stale value. E.g. On Fridays, income will be calculated for Friday, and the same value (which is now stale) will be used for Saturday and Sunday. Therefore the income value will represent 3 days.

  • Split evenly before and after

2 days will be accrued on a Friday and 2 days on a Monday. Friday's accrual is for Friday and Saturday, and Monday's accrual is for Sunday and Monday, i.e. the non-working days are split between Friday and Monday.

Note: Working days / Non working days are defined on the calendar associated with the oversight entity.

 Holdings Tab

The Security Repricing attributes are used when calculating the holdings component of an Expected NAV for an entity. An entity's holdings will be revalued using the current day price from the U175: Security Prices worktable. The security price run type selected in this attribute, is matched to the same security price run type on the U175 worktable to obtain the correct price.

Where fair value functionality is required, the fair value attribute must be selected, and the Fair Value entity selected, in order to specify which values should be retrieved from the U107: Price Component Adjustments worktable. For further details, refer to [ERM003] Holdings - Security Repricing

 

 Associating the ‘Expecting Pricing Methodology’ entity to an Oversight entity

Once the Expected Pricing Methodology entity is configured, it can be associated to the oversight entity via the Expected Pricing Methodology attribute under the Expected Pricing tab.

This attribute is used during the Calculations phase of the Expected Pricing process. There are several different Expected Pricing Methodologies, which are explained in the Expected Return Calculation Methodology section.

 

 Expected Return Calculation Methodology entity

The Expected Return Calculation Methodology entity is associated to a rules file, where the formula for the calculation is stored. The formulas cannot be seen, however the details of each formula, are detailed in Expected Return Calculation Methodology section.
