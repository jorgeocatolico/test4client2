Exceptions

pControl facilitates exception processing. The SLA Monitor can be configured in System Administration to alert users to processes which require intervention and indicate time frames for turnaround on such exceptions. The SLA Monitor is driven by the configured status stream described below.

The exceptions view is designed to display all exceptions generated as a result of a breach of validation tolerance. By default, exceptions are displayed under their configured states, however this, along with other display criteria can be changed.

Note

If accessing the exceptions page via the SLA Monitor, the exceptions list may already be filtered.

Understanding Exceptions

When a process encounters an exception, it will initially be listed in the 'Not Validated' state on the exceptions dashboard. From there, it can be transitioned to a different state on the status stream. Exceptions with the same criteria, e.g. breach of the same validation check for the same asset, or exceptions assigned to the same team for review can be collectively reviewed and cleared simultaneously, or assigned to a different team for further review. Once the checking process for all exceptions related to a control record has been completed, pControl will transition all exceptions to ‘Checked’, which will allow the control record to continue along its status stream life cycle.

Exceptions Status Streams

A status stream is a predefined flow of states and transitions. The Exceptions Status Stream, (also known as the 'Override Event' status stream in System Administration), is used to manage the life cycle of an exception related to a control record.

Further detail is provided in Exceptions Status Stream states and transitions.

 

 Filtering Exceptions by Status, Severity and Asset Code

Before filtering, all exceptions are loaded and all exception events can be viewed by scrolling, without a limit on the number of rows.

Selecting the <More> filter options on the primary filter provides the option to choose which exception events to display in the panel below and using the <Group By> options, manages how the events are grouped. Filtering by Severity, for example, means that transitioning exceptions can be prioritised accordingly. Filtering by status accelerates the ability to complete multiple transitions at the same time.

 Tolerance Framework

The tolerance framework is a key part of the overall process configuration in pControl and is used to establish and maintain the tolerances that will be used by all validations for all entities. The tolerance framework is configurable, allowing tolerances to be maintained right down to the entity level and also for groups of entities or for all entities in pControl.

Tolerances can be set at different levels for the same validation so that different levels of exception are raised depending on the severity of a breach. The severity level of these items is seen in the status messages as well as the exception event. Only certain tolerance levels will result in a breach thus causing an exception to be generated. This is shown in the table below by the 'Create Exception Event (Y/N)' column.

The different tolerance levels that can be set for a validation are as follows:

Tolerance Status

Create Exception Event

Description

OK

N

Presents an ‘OK’ message but no exception event and this message will not prevent a control record from being approved.

Warning

N

Presents a ‘Warning’ message but no exception event such that this message will not prevent a control record from being approved.

Alert

Y

Presents an ‘Alert’ message and creates an exception event such that the control record cannot be approved until this issue is resolved, i.e., the exception event is cleared or data changed and the process re-run.

Critical

 Y

Presents a ‘Critical’ message and creates an exception event and the control record cannot be approved until this issue is resolved, i.e., the exception event is cleared or data changed and the process re-run.

DC Error

 Y

Presents a ‘DC Error’ (Data Consistency Error) and creates an exception event such that the control record cannot be approved until this issue is resolved i.e., the exception event is cleared or data changed and the process re-run.

Exception

 Y

Presents an ‘Exception’ message and creates an exception event such that the control record cannot be approved until this issue is resolved, i.e., the exception event is cleared or data changed and the process re-run.

L2 Exception

 Y

Presents a ‘L2 Exception’ message and creates an exception event such that the control record cannot be approved until this issue is resolved, i.e., the exception event is cleared however a user with L2 security access must then transition the status stream to Pending Approval or data changed and the process re-run.

L3 Exception

 Y

Presents a ‘L3 Exception’ message and creates an exception event such that the control record cannot be approved until this issue is resolved, i.e., the exception event is cleared however a user with L3 security access must then transition the status stream to ‘Pending Approval’ or data changed and the process re-run.

Note: If a combination of L2 and L3 Exception messages is presented the exceptions can be cleared however both L2 and L3 users will be required to transition the status stream (e.g. transition by a user with L2 security access with further transition by a user with L3 security access).

The tolerances are also managed in a hierarchy such that if a specific entity is not listed in the tolerance table, it will look for an entry for any parent entity of the entity in the library tree. If there is no parent entity of the entity in the table, it will use the DEFAULT tolerance for the validation.

In this way, the hierarchy that is used to search for the relevant tolerance to use for a specific entity validation is as follows:

Entity - has the specific entity being processed been set up in the ‘Entity Code’ column of the L200: Tolerances worktable against this validation entity? If so, use it, otherwise move on.

Direct Parent Entity of Entity – has the direct parent entity (in the library tree) of the specific entity being processed been set up in the ‘Entity Code’ column of the L200: Tolerances worktable against this validation entity? If so, use it, otherwise move on.

Next Level Parent Entity of Entity (up library tree to Company Entity level) - has the next level parent entity (in the library tree) of the specific entity being processed been set up in the ‘Entity Code’ column of the L200: Tolerances worktable against this validation entity? If so, use it, otherwise move on.

DEFAULT – if none of the three scenarios exist, use the default tolerance settings for the validation entity.

Example

The entries in the POV010_PRICE_MVMT_BMARK table can be interpreted as follows.

This is the same logical sequence that would be used when running validations against an entity:

  1. Entity - if the entity on which the validation is being run is BALANCED_FUND, then the first two rows of the L200 will be used, If the check value of ACC004 validation check,
  2. Deviates from the tolerance by less than 50 basis points the validation passes.
  3. Deviates from the tolerance by 50 basis points or more, and less than 500 basis points, an ‘L2 Exception’ is issued.
  4. Deviates from the tolerance by more than 500 basis points, an 'L3 Exception' is issued.
  5. Parent Entity of Entity – if the entity being processed is located underneath BALANCED_FUND in the library tree, (i.e. is a child of BALANCED_FUND), than the above tolerance values will be used to compare the results of ACC004 validation check.
  6. DEFAULT - if the entity being processed is not BALANCED_FUND and it does not have BALANCED_FUND as a parent at any level of the library tree, then the tolerances with 'Entity Code' = 'DEFAULT' will be used to compare the results of ACC004 validation check. The tolerances will be interpreted as follows:
  7. If the result of the validation check
  8. deviates by less than 50 basis points, the validation passes.
  9. deviates by more than 50 bp and less than 100 bp, an 'Exception' is raised.
  10. deviates by more than 100 bp and less than 200 bp, a 'L2 Exception' is raised.
  11. deviates by more than 200 bp, a 'L3 Exception' is raised.
 Validations and Tolerance Configuration

Different validations require different tolerance configuration. Some of the different validations and their tolerance configurations are explained below.

Tolerance Type

Description

L200: Tolerance

The L200 tolerance values are used for comparison, and also generates the message (and exceptions depending on the level of tolerance status) displayed when a tolerance is breached.

i.e. [BMK004_MVT_PCT] where the validation tests the movement in a benchmark from the previous daily valuation point against a specified tolerance.

L200: Message Only

No numeric values used for comparison, i.e. a true / false check, and the L200 Tolerance status generates the message (and exceptions depending on the level of tolerance status) displayed when check fails.

i.e. [HLD001] Zero Cost Check which will fail the test if the cost (local or base) of the holding is zero.

No tolerance required

No tolerance is required in the L200, as the rules capture all requirements, i.e.[VC010] - Check if Entity is Set to Process

If the ‘Include in Processing’ attribute is not selected the test fails and processing stops with the control record moved to ‘Unable to Price’.

No tolerance required (Value Add)

No tolerance is required as it is not a true ‘validation’ but a ‘value add’ calculation / function i.e. VA011: Generate Dynamic Benchmark.

 

Using the Exceptions Functionality

Transitioning or Clearing Exceptions

Exceptions are transitioned by selecting the ellipsis followed by the state to transition to. An individual or group of exceptions with the status of 'Proposed Validated' indicates that the analysis of the exception is complete and is ready for authorisation by the second user. Where no authorisation is required, the individual or group of exceptions can be transitioned straight to 'Validated'.

Further information relating to an exception event can be accessed via Analysis Views and Fund Focus Views.

To transition to a new state or clear the exception:

  1. Select the ellipsis and choose the transition state from the options available i.e. Carried Forward Analysis, Pending Data Remediation or Proposed Validated.

  2. In some instances, a comment will be required before the transition can be completed.

     

  1. If saved, the exception state will update accordingly.
Note

The state transitions are driven by the Override Event status stream which is defined in System Administration.

If the selected transition state means the record is directed to another user, the second user can review the changes made and if they agree, can transition the exception to 'Proposed Validated'. If they disagree, the exception can be moved to 'Carried Forward Analysis' or 'Pending Remediation'. However only users with the highest level of user access can make this transition.

Once all exceptions related to a control record are in a ‘Validated’ state, the control record will automatically progress to the next state on the control record approval status stream.

If all exceptions related to a control record are cleared and they reach the ‘checked’ state, it is not possible for the exception to be transitioned back to any of the earlier states in the status stream. Therefore, if an exception reaches the ‘checked’ state, in error, i.e. the first user and alternate user both made a mistake, the process will need to be re-run so as to recreate the exceptions.

When transitioning multiple records, selecting the ellipsis will only display the available states common to all records selected. If there are no common options available, a message will indicate this. If several exceptions are transitioned simultaneously, they will all have the same bulk id reference number.

Note

If an exception is caused by an error in the underlying data, and new data needs to be loaded and the process will need to be re-run in order to generate a process without exceptions.

 

 Recurring Exceptions and Using the History Function

Recurring exceptions can be reviewed via the Analysis Views. The History tab provides details of all exceptions for the selected asset and what action was taken each time. If there are a number of exception events which are similar, it is an opportunity to review whether or not tolerance levels should be adjusted and subsequently the number of the exception events requiring intervention will be reduced.

  1. For the exception event, select the ellipsis, followed by Analysis.

  2. Select the <History> tab to view the count of how often the exception has occurred in the last week. There is also an option to view the count for the Last Month.

 Asset Level Validation

The Asset Level Validation displays all validations that have been performed for the selected asset or account, for example, Holding Validation, Transaction Validation or Account Validation.

The charts show the movement in holdings, price, value for example, and indicates what has changed which has contributed to the exception event.

Detail Tabs available on the Asset Level Validation Analysis View

Business

Details on the asset holding including price, weight in fund, asset return.

Lists any exceptions related to the holding.

Provides an indication of the movement between today and yesterday.

Transactions

The A05: Transaction Validation worktable lists of transactions performed during the selected date.

Security Prices

The U175: Security Prices worktable lists of asset prices for all assets held by one or many funds.

Typically sourced from a source independent to the TPA.

Security Master

The U180: Security Master worktable holds static data related to the assets held by one or many funds. Data is used by some of the expected pricing formulas.

Validations

Displays any other validations that have been performed on the same asset for this fund for the current run only.

Related Funds

Other funds that have encountered this exception for this asset.

Validation History

All occurrences of this validation for the same criteria.

 

 Fund Level Validation

The Fund Level Validation displays all validations that have been performed for the selected fund, for example a Product Validation or Category Validation.

 

Detail Tabs available on the Asset Level Validation Analysis View

Business

Details on the asset holding including price, weight in fund, asset return.

Lists any exceptions related to the holding.

Provides an indication of the movement between today and yesterday.

NAV Movement

Provides a breakdown of the NAV by category i.e. EQ, FI, Cash.

Comparison of the breakdown vs previous pricing point.

Also referred to as VMAS.

Account Movement

Displays NAV components based on GL account values.

Includes benchmark values, unit prices and NAV values.

Also referred to as AMAS.

Share Class Impacts

Class comparison analysis that provides a view across classes within the same fund.

Also referred to as CPAS.

Oversight Movement

Oversight movement analysis provides a comparison of expected vs oversight price records and components of those pricing processes.

Also referred to as OMAS.

Distribution Yield

Provides a breakdown of the components of the distribution and compares the value with the prior distribution.

Also referred to as DYAS.

Related Funds

Other funds that have encountered this exception for this asset.

Validation History

All occurrences of this validation on the current date for this fund and asset.

History

All exceptions that have occurred for this fund and price run type over the past week and month.
