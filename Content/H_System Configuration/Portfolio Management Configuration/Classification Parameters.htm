Classification Parameters

Classifications enable the display of proposed investments via different classifications, such as, but not limited to: asset class, or economic sector groupings, custodian or country of origin. Classification Parameters, sometimes referred to as 'CALP', are validation checks which can be performed against these user defined classifications to ensure compliance with various defined limits, or business rules. For example, International Equities could have a classification parameter to ensure its weight within the plan is between 20% to 30% of the total plan value; or its weight within all international sectors is between 70% to 80%.

Classification parameters are considered a 'set' as all the parameters are stored for the same start date, unlike Investment parameters. Where changes are made to one or more parameters within the same set, the changes will invoke one pending change rather than individual pending changes.

Up to five Classification Parameters can be configured for each Classification set. The validation checks are performed against the classification plan during the 'Approval' phase of the Cash Allocation Process. Classification values which are outside of the tolerance / bound will be indicated via hard or soft warnings.

Classification Parameters

Adding classification parameters to a classification plan is the last step in Setting up a Plan and Classifications and can be accessed via the menu navigation: Configuration > Classification Parameters.

Note

Only users with the appropriate security permissions will be able to view, edit or authorise classification parameters.


Tolerances and bounds

If the value of an asset or sector moves outside of the limits, the order may not be able to be approved, depending on whether a 'hard' or 'soft' warning is detected. There are two limits defined within pControl, 'tolerances' and 'bounds':

  • Tolerances are the percentage movements away from a target where the portfolio is considered outside tolerance.
  • Bounds are absolute upper and lower limit where the portfolio is considered out of bounds.

Classification breaches

When compliance checks are run and there is a classification sector breach, any proposed orders that are related to that classification sector are moved to an exception state. These proposed orders are filtered to the top of the Proposed Transactions panel.

For aggregate and derived sectors, all proposed orders (i.e. the children) that belong to the sector are transitioned to the relevant hard or soft exception state.

For proposed orders that have a soft investment compliance breach and a hard classification sector breach, then the proposed order transitions to a hard exception.

For proposed orders that have a hard investment compliance breach and a soft classification sector breach, then the proposed order transitions to a hard exception.

For proposed orders that have various hard or soft sector breaches, then the proposed orders transition to hard exceptions.

All proposed orders that have been transitioned to an exception state, the processing notes are updated with all sector breaches.

Mid-level originator breaches

Mid-level originators do not generate orders, therefore any investment or classification breaches that take place will instigate an exception on any of its children and the mid-level originator will remain in its 'Not Calculated' state. Like other breaches, the processing notes will be updated with details of the sector or investment that breached.

Liquidity and exposure assets

Liquidity and exposure assets also do not generate orders and are directly transitioned to approve. To prevent all proposed orders from being transitioned to the approval state, it is necessary that other assets are transitioned to an exception.

For a batch of proposed orders (for the same plan, date and CAR process), if there are classification compliance breaches which relate to liquidity or exposure assets, then the following sets out how to identify which assets are to be transitioned in place of the liquidity and exposure assets.

  • Other asset will be transitioned to an exception state - in the case where at least one proposed order will be transitioned to an exception state, then there is no need to identify any other assets to transition in place of the liquidity or exposure assets (assuming there is at least one proposed order exception that does not relate to a liquidity or exposure asset).
  • Transition a sibling asset - in the case where no other proposed order will be transitioned to an exception state then the siblings of the liquidity or exposure orders will be transitioned to an exception state in place of the liquidity or exposure assets. Sibling assets are assets who have the same parent aggregate sector as the related liquidity or exposure sector. If all siblings are liquidity or exposure assets then this is taken as if there are no siblings).
  • Transition of all assets - in the case where no other proposed order will be transitioned to an exception state and no siblings of the liquidity or exposure assets exist then all order will be transitioned to an exception state in place of the liquidity or exposure assets.

 

Modifying Classification Parameters

Pending changes are considered as a set and therefore require only one transition to the next state. When classification parameters are authorised, they are included in that version of the classification structure. When classification parameters are rejected, they are excluded in the classification parameters that are versioned.

For example, a Portfolio Analyst makes changes to the Schroders USD Equities Fund's lower tolerance and bound; and increases the upper tolerance and bound of the BlackRock USD Equities Fund. The changes are made in the table itself and all edits have a corresponding pending change card. Once validated and saved, the transition menu will become available and the changes can be assigned for review.

Where any parameters fall outside of the validations, a hard or soft warning will display in the Validations panel. It is still possible to undo the change after saving by selecting the trash can on the pending change panel.

Validations panel

Records can be validated at any time by selecting <Validate>. When changes are made and the change breaches validation, a warning will display as shown below. In some instances, it may be possible to override the warning, however this will be dependent on permissions and configuration set out in Compliance Checks.

Selecting the validation in the panel, will highlight the corresponding value in the table that the validation is referring to.

Authorising Classification Parameters

The Portfolio Manager, the reviewer, is able to view the list of changes made in the Pending Changes panel and can navigate through each of the changes either by selecting the cards themselves, or by selecting the pending change arrows at the top of the pending change panel.

Each change will display in the table and will be highlighted when selected on the pending change panel. The pending changes are treated as a set therefore they require only one transition to the next state on the status stream. Once reviewed, the Portfolio Manager can select the appropriate transition, for example, 'Requires Modification' or 'Authorised'.

Show original values

If <Show original values> is selected, the table will include the rows showing only the values that have been edited.

Transitioning in the Classification Plans view

The Portfolio Manager may also use the ellipsis on the classification plan to transition the pending changes but will not be able to view the detail of the changes except where a Classification Set is to be deleted, as shown below.

 

 Reviewing the Changes on the Pending Changes Dashboard

The Portfolio Manager is able to view changes to Classification Parameters via the Pending Changes Dashboard. Changes are due by SLA time and by state, i.e. Ready to Authorise.

Selecting <Review> navigates to the Classification Parameters record where the Portfolio Manager can review the pending changes. Once the pending change workflow has been completed, i.e. approved, rejected, the record will no longer appear on the Pending Changes Dashboard.
