Refactor Weights for Transaction Minimum and Maximum Constraints

pControl allows for the refactoring of weights where there are transaction minimum and maximum breaches, by utilising the attribute on the Cash Allocation - Calculations tab, as shown below.

If set to true, any asset that has hit a transaction minimum or maximum will be treated as 'Excluded'. This means current and target weights will be refactored, and a new set of proposed orders will be generated. If set to false, whatever cannot be allocated due to the minimum/maximum will be treated as spare cash.

The supporting Transaction Maximum Refactor Methodology provides two options. Where 'Allocation up to the Maximum' is selected, pControl will treat the asset as excluded, but first allocate up to the maximum amount. If 'Do Not Allocate up to the Maximum' is selected, nothing at all is allocated to the asset.

Where transaction minimum constraints are not met, the assets that do not meet the constraints will have their target and actual weights set to 0.00 and the weights of the tradeable assets within the portfolio or target group will be refactored. The total cash is reallocated based on the refactored target and actual weights. This is demonstrated in the examples below.

In cases where the asset with the transaction minimum/maximum constraint is part of a target group, the refactoring and reallocation will occur within the target group (as in the case of excluded assets that are part of a target group). If reallocation cannot occur within the target group, then the cash will either remain in liquidity or be allocated as per the spare cash methodology defined on the entity.

In rebalance scenarios where there is only one (or none) tradeable asset due to the transaction minimum/maximum constraints, then no rebalance will occur i.e. proposed orders for 0.00.

Transaction Minimum - Examples

In the case of a transaction minimum, the entire amount of the suggested proposed order goes to either liquidity or spare cash. The Refactor Weights for Constraints attributes provides the option to treat spare cash as a result of a transaction minimum as an 'Excluded Asset' for a given entity.

Where the transaction minimum constraints are not met, the assets that do not meet the transaction minimum constraints will have their target and actual weights set to 0.00 and the weights of the tradeable assets within the portfolio/target group will be refactored. The total cash would then be reallocated based upon the refactored target and actual weights.

When the 'Refactor Weights for Transaction Minimum /Maximum Constraints' is set to true, any proposed order that does not meet the transaction minimum/maximum criteria will not go into spare cash or liquidity. The assets will be treated as an 'Excluded Asset', and pControl will refactor the target and actual weights and then reallocate across the portfolio/target group.

 Transaction Maximum - Allocate up to the maximum examples

Where the attribute 'Refactor Weights for Transaction Minimum/Maximum Constraints' is set to true and the Transaction Maximum Refactor Methodology is set to 'Allocate up to the maximum', pControl will create proposed orders up to the maximum amount as shown in the example. The target and actual weights will be refactored and excess cash over the maximum will be reallocated across the tradeable assets within the portfolio/target group.

Refactoring of target/actual weights in this scenario will occur in the following manner:

  • Refactored Target calculation will set the target of asset exceeding maximum to 0.00.
  • Refactored Weight calculation will set the weight of the asset exceeding maximum to 0.00.
  • Refactored Start Investment Holding value of the asset exceeding maximum must include the value of the proposed order up to the maximum.
  • Liquidity value used in the Refactoring Target/Weights must exclude the value of proposed order up to the maximum.

 Transaction Minimum and Maximum - Rebalancing examples

If only one asset is tradeable in a rebalance scenario due to minimum/maximum constraints, proposed orders should not be generated therefore no rebalance is performed. If after the final iteration of refactoring weights due to transaction minimum/maximum constraints, there are not at least two tradeable assets, then no rebalance occurs.

 Refactoring Tolerances and Bounds

Where [PAP030] Accept/Reject Refactored Tolerances and Bounds is selected, the tolerances and bounds will be refactored along with the target and actual weights for each iteration i.e. each time a target weight is refactored as a result of the transaction minimum/maximum constraints.
