Cash Allocation and Rebalancing Methodologies

Cash Allocation methodologies supported in pControl include:

  • Straight Percentage
  • Zero Cash To Allocate
  • Towards Target
  • Box Management

Rebalancing methodologies supported by pControl include:

  • Rebalance to Target
  • Partial Rebalance to Percent Target

 

The methodology to be used for Cash Allocation and Rebalancing processes is specified through the ‘Allocation methodology’ attribute on the fund. The Cash Allocation attribute is shown below. For rebalancing, navigate to the Rebalancing process in the left hand column and select the Calculations phase.

Cash Allocation Methodologies

Straight Percentage methodology

 

Business Description

The cash available to allocate (liquidity balance) is divided up amongst the investments specified by the Allocation Parameters (ALPs). This creates a proposed order for each investment (application or redemption). This value is written into the ‘Ideal Order’ field on the Proposed Order.

Data Requirements
  •  

Processing Requirements

For each ALP, the straight percentage calculation is:

$Ideal\ Order\ =\ Cash\ available\ to\ allocate\ \times \left ( Target\ Weight\ \%\ or\ Current\ Weight\ \%\ for\ the\ asset \right )$

Where Orders Criteria grouping is selected on the entity, the calculation is:

$Ideal\ Order\ =\ Cash\ available\ to\ allocate\ for\ order\ criteria\ group\ \times \left ( Target\ Weight\ \%\ or\ Current\ Weight\ \%\ for\ the\ asset \right )$

The entity attribute Straight Percentage Weight Method determines if Target or Current weight from the ALPs is used in the calculation.

Entity Data Attributes Used

 Straight Percentage Methodology

The following attributes are used by all cash allocation methodologies (unless otherwise specified):

  • Allocation Methodology
    • Select methodology.
  • Allocation Grouping
    • The setting of this attribute will have an impact on the price date on the proposed order and subsequent actual order. This applies to both cash allocation and rebalancing runs.

    Effective Date

    The effective date allocation grouping will create orders from the fund with a price date equal to the effective date. This is the same as normal cash allocation, i.e. no deferral of cash allocation orders is required.

    Forward

    The forward allocation grouping will create orders from the fund with a price date equal to the greatest deferral (notice) date on the bottom-most-level asset invested in. The settlement date on the proposed order will equal to the ‘Value Date’ configured on the asset entity level.

    Historic

    The historic allocation grouping will create orders from the fund with a price date equal to the last approved price date of the asset. This is similar to standard cash allocation, i.e. no deferral of cash allocation orders is required, although an historic price may be used if historic allocation grouping is specified.

    Forward and Historic

    If ‘Forward and Historic’ allocation grouping is selected, then two cashflows groups from the fund will be created – one containing orders with a forward (deferred price) date, and another with a historic (last approved price) date.

    Orders Criteria

    The ‘Orders Criteria’ allocation grouping uses the allocation criteria in the Orders Criteria attribute table to determine how the incoming cashflows are processed. This ensures that the relevant price date, transaction type or price type for any cashflows received is applied consistently throughout the investment structure.

    Blank

    If left blank, pControl will default to 'Effective Date'.

  • Straight Percentage Weight Method
    •   Determines whether the Target or Current weight from the ALPs is used in the allocation calculations.

    Target Weight

    pControl will use the value set in ALP

    Current Weight

    pControl will use the Asset Allocation
    Blank

    pControl will default to use Target Weight.

      
 Straight Percentage Methodology with Orders Criteria Grouping

The price or ‘value’ date, transaction type or price type for any cashflows received can be applied to proposed orders generated throughout the investment structure. This means that together with a ‘straight percentage’ method, the cashflows can be further grouped by the specified criteria. The entity attributes are as follows:

  • Allocation Methodology
    • Straight Percentage
  • Allocation Grouping
    • Orders Criteria Grouping
  • Orders Criteria
    •  Ensures that the relevant price or 'value' date, transaction type or price type for any cashflows received is applied throughout the investment structure.

    Price Date

    Refers to the different value dates populated in the 'Price Date' field in the U110: Cashflows and Orders worktable with the same price date, will be grouped together.

    Price Type

    Refers to the different price type populated in the ‘Price Type’ field in the U110: Cashflows and Orders worktable. Cashflows with the same price type (e.g. buy price, sell price) will be grouped together.

    Transaction Type

    Refers to the transaction types populated in the ‘Tran Type’ field in the U110: Cashflows and Orders worktable, together with the transaction types that have been configured in the ‘Group in Cash Allocation’ field in the L250: Transaction Flow Configuration worktable. Cashflows with the same transaction type will be grouped together.

    For example, if Price Date is selected, the cash available to allocate is divided by the price date. The value of orders loaded in each price date group is then processed individually with proposed orders made for each investment according to both the ALP weights and the price date group. Where price date is specified for grouping, a proposed order will be created for each investment with the price date equal to the effective date (application or redemption). The value is then written into the Ideal Order field on the Proposed Order with the price date equal to the effective date.

     

 

 
Zero Cash to Allocate methodology

 

Business Description

This methodology allows users to have the option not to allocate cash to funds when a cash allocation and rebalancing process is run, even though target weights and liquidity have been set up in an ALPs. Proposed orders with zero value will be created, and this provides the opportunity to adjust the proposed order amounts, or to leave the values at zero.

Data Requirements
  •  

Processing Requirements

The ‘Zero Cash to Allocate’ method will simply zero out the ideal order value on the proposed orders, even when there is cash to allocate. The starting position, available cash, etc. remain viewable on the proposed order.

Entity Data Attributes Used
  • nil

 
Towards Target methodology

 

Business Description

This methodology allocates cash towards target based on proportional difference to target amounts. Positive cashflows are allocated proportionally to assets underweight, whereas negative cashflows are allocated proportionally to assets that are overweight. Positive cashflows are never allocated to overweight assets and vice versa.

Data Requirements
  •  

Processing Requirements

Where the liquidity balance is positive, proposed orders for underweight assets will be calculated proportionally based on the difference to target amounts. The value of the liquidity balance is always included in the calculation of the start weights (and thus the difference to target amounts). The underweight asset that is farthest from target will get the largest allocation of cash.

Note, this is NOT a ranked methodology in which the asset that is farthest from target is first brought all the way to target, and so on.

Allocation of cash is proportional to all eligible assets based upon difference to target amounts. Cash that cannot be allocated (in the event that all assets are at their target) will either remain in liquidity or be allocated as per the spare cash methodology defined on the entity.

The same methodology and behaviour applies for negative cashflows.

If Cash to Allocate > 0 and Difference to Target < 0, then ideal order =

$Cashflow\ Amount\ \times \large \left ( \frac{ Difference\ to\ Target\ for\ ALR }{Total\ Difference\ to\ Target\ for\ all\ Unders} \right )$

If Cash to Allocate < 0 and Difference to Target > 0, then ideal order =

$Cashflow\ Amount\ \times \large \left ( \frac{ Difference\ to\ Target\ for\ ALR }{Total\ Difference\ to\ Target\ for\ all\ Overs} \right )$

Entity Data Attributes Used
  • nil

The ‘Box Management’ methodology in the pControl cash allocation product supports the notion of using a manager’s box within an investment structure to effectively trade on behalf of the manager.

Rebalancing Methodologies

Rebalance to Target methodology

 

Business Description

Rebalancing is usually performed to re-weight any investment holdings that are out of tolerance. In pControl, this process calculates and creates proposed orders in the same way as Cash Allocation.

For each Allocation Parameter (ALP), pControl checks whether the starting position in each asset specified by the ALP is outside of the tolerances that have been set, i.e. the following test is performed.
Data Requirements
  •  

Processing Requirements

A rebalance will be initiated if any of the starting positions are outside of tolerance, including liquidity. If there is an amount in cash when the rebalancing runs, and it is outside of the upper tolerance, then the rebalance will generate a proposed order to bring the holding to within the upper limit.

If $Start\ Weight\ \% < \left ( TW\ -\ Lower\ Tolerance \right )$ OR $Start\ Weight\ \% >\ \left ( TW\ +\ Upper\ Tolerance \right )$

then, propose a rebalancing order.

If $\left ( TW\ -\ Lower\ Tolerance \right ) \leqslant\ Start\ Weight\ \% \leqslant\ \left(TW\ +\ Upper\ Tolerance \right)$

then the investment is within tolerance, and a rebalance will not be proposed.

For each investment that has been identified as outside of tolerance, the calculation is as follows:

$Ideal\ Order\ =\ \left (TW\ \times Total\ Investment\ Value\ of\ the\ fund \right )\ -\ Starting\ value\ of\ investment\ in\ the\ asset$

Ad-hoc Rebalancing

Ad-hoc Rebalancing functionality allows funds to request rebalancing outside of their standard schedule rebalancing periods, with special allocation parameters used, if required, to override the default strategy and tolerances. Ad-hoc rebalancing is supported by both Rebalance to Target and Partial Rebalance to Percent Target methodologies. Jen to insert link here.

A Rebalancing Calendar can be defined by using a ‘Calendar Schedule Entity’. The calendar schedule entity contains the calendar which determines the scheduled days when ad-hoc rebalancing should be allowed to run. For example, where a scheduled rebalance occurs on a public holiday, the default may move this rebalancing event to the next working day, however ad-hoc rebalancing can override this event, so that the rebalancing process takes place on the public holiday.

 

 Miss a Scheduled Rebalance

If selected, the rebalancing process will not run if an ad-hoc rebalance process has been run since the last rebalancing process.

 

 Nudge Rebalancing (Glide Path)

The daily nudge method uses a series of short-term daily rebalance events over a defined period to gradually move from the original weighting to the new target allocations. For the period between receipt of the request for change and the end date, predefined target allocations (ALPs) will be used to achieve a gradual switch of assets from one fund to another each day until the target is achieved. This is known as the ‘glide path’.

At the end of the period, the fund may revert to use the ALPs that were in place prior to the nudge rebalancing occurring, or the target ALPs can be reset to ensure that the new target allocations are used.

There are two nudge rebalancing methods:

Cash Flow Method

When the glide path method is ‘Cash Flow’, the investment manager is able to specify a target asset allocation that they want to move to, and a timescale (glide path) over which they want to achieve this transition.

When the current (Rebalancing) Allocation Methodology is ‘Straight Percentage’ and the rebalancing falls within the glide path period, pControl will change the allocation methodology from ‘Straight Percentage’ to ‘Towards Target’. Therefore, on a daily basis, the cash allocation process will utilise the net new money and allocate to either the most underweight or most overweight depending on the cash flow for the day.

The Towards Target methodology will check if the new holding is overweight or underweight, and readjust the proposed order amount by giving a proportionate amount to the other under/overweight ALPs

On the last day of the glide path period, the rebalancing process will rebalance using the ALPs that are effective on the glide path end date.

 

 Rebalancing Method

When the glide path method is ‘Rebalancing’, and it is in a glide path period, the investment manager is able to specify a target asset allocation that they want to move to and a timescale over which they want to achieve this transition.

On a daily basis, the cash allocation process will perform the necessary rebalancing transactions to take the fund back to the calculated target asset allocations. Tolerances may be applied on a daily basis but it is likely that they will be set to zero during the duration of the rebalance, to allow a rebalance to occur each day.

Rebalancing will occur to the calculated glide path ALPs for each day in the period.

 

 

 Calculating Glide Path ALPs

The ALPs that are used during the glide path period are calculated for each day in the period and depend on the glide path method that is used, i.e. Cash Flow or Rebalancing.

When it is NOT the last day in the glide period and irrelevant of cashflow or rebalancing method selected, the calculation is:

$TW\ =\ Start\ Date\ TW\ +\ \frac{ \large \left ( End\ Date\ TW -\ Start\ Date\ TW \right ) }{ \left ( \large Total\ Duration\ Glide\ Path\ \times Current\ Day\ in\ Glide\ Path \right ) }$

Where:

  • TW = Target Weight
  • Start Date TW is the ALP attribute effective at the start date of the nudge period
  • End Date TW is the ALP attribute effective at the end date of the nudge period
  • Total Duration Glide Path is the Total number of days in the nudge period inclusive of the start and end date
  • Current Day in Glide Path is the Current day number in the period.
  •  

When it IS the last day in the glide period, and the Cash Flow method is used:

$TW\ =\ End\ Date\ TW$

where the End Date TW is the ALP attribute effective at the End Date of the nudge period.

Partial Rebalance to Percent Target methodology - ready for review

 

Business Description

The ‘Partial Rebalance to % Target’ methodology checks each Allocation Parameter (ALP) to determine whether the starting position into the asset specified by the ALP is outside of the tolerances that have been set.
Data Requirements
  •  

Processing Requirements

If any investment is identified as being outside of tolerance, an ‘Ideal Order’ is calculated for all investments as follows:

$ \left ( Partial\ Rebalance\ Weight\ in\ asset\ \times Total\ Investment\ Value\ of\ the\ fund\ \right ) -\ Starting\ Value\ of\ the\ Investment\ in\ the\ asset$

Where, Partial Rebalance Weight in asset is calculated as:

$Percent\ Towards\ Target\ \times Target\ Weight\ in\ asset\ + \left ( 1\ -\ Percent\ Towards\ Target \right ) \times\ Starting\ Position$

Where, Starting Position is the % value.
