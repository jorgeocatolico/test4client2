Cash Allocation Functions

Run Compliance Rules

Processing Phase

Approval Validations

Description

This function permits the execution of specified compliance rules on all proposed orders before they are approved. The rules are executed in the order listed in the table.

  • If a proposed order fails these checks, it is put into a state of ‘Exception’.
  • If a proposed order passes the compliance rules then it is written as a proposed order and;
  • automatically approved if that option has been configured or;
  • will wait to be transitioned to approved by the users

Each compliance rule passes back a true (pass) or false (fail) value and if any of the compliance rules fail, the proposed order will be written with an ‘Exception’ status.

The ‘Processing Notes’ field on the proposed order will record details of the compliance rules that have failed so the users can understand why and take action.

There are four courses of action available when faced with a proposed order in an Exception state:

  • Override the Exception –transition the proposed order state to ‘Overridden’, and enter a comment explaining why the compliance rules were overridden. Then transition the proposed order to an ‘Approved’ state which triggers an actual order to be created. The override comments will be displayed in the ‘Processing Notes’ field.
  • Cancel the Proposed Order - transition the proposed order to ‘Pending Cancel’ which cancels the proposed orders with no actual orders created. This is a similar end result to cancelling the process manually. Pending cancel will end the cash allocation process without firing the completion EDNs and therefore will not trigger interfund processes. The cash allocation process can then be re-run at certain level of the interfund.

Note: If any of the proposed orders are set to Pending Cancel, all proposed orders will be transition to Cancelled, and the Master proposed order is transition to Not Started.

  • Reject the Proposed Order –transition the proposed order to ‘Pending Rejection’ which rejects the proposed orders with no actual orders created however this will mark the cash allocation processed as completed and trigger the next level of cash allocation process to start.
  • Run Again –transition the process to ‘Run Again’ after amending the proposed order (user) amount (via ‘Maintain Proposed Orders’ screen) and re-run compliance rules. This will force the compliance rules to be re-executed and a new result generated which could be another exception if they fail, or an approved proposed order if they pass.

The ‘Comment’ entered by the user is recorded in the process notes field on the proposed order along with the reasons that the compliance rules failed.

Function type

Process Template

Entity data attributes used

Compliance Rules – this attribute table lists the compliance rules that need to be executed by this entity if the ‘Run Compliance Rules’ function is selected on the process template

In this table, there are three fields to be populated:

  • Rule Code – The Cash Allocation ‘Product’ compliance rules include:
  • Ensure all proposed orders are Approved or Rejected
  • Check confirmation date is a valid trade date
  • Check if any external assets have missed cut off
  • Check interfunding trading

The entity attributes used are dependent on the specific compliance rule. The compliance rules are detailed in the sections below.

  • Comment – This is a free text field, the comment populated will appear in the Value field of the processing status.
  • Apply Rule To – ‘Entity’ or ‘Proposed Order’ can be selected based on the individual compliance rules selected
  • Entity – Selected when compliance rules are related to entity attributes
  • Proposed Order – Selected when compliance rules apply to the actual proposed orders

Tolerance type

No tolerance required

Pre-requisite functions

Proposed Orders

Outputs

Approved, Rejected or Exception Proposed Orders

 

Ensure all proposed orders are Approved or Rejected

Processing Phase

Approval Validations

Description

This function ensures that all proposed orders are either ‘Approved’ or ‘Rejected’ i.e. treated the same way. This is to ensure that a user does not reject one proposed order and approve another proposed order from within the same cash allocation /rebalance process.

  • If any proposed orders have been transitioned to a state of ‘Pending Rejection’ and there are proposed orders present in a state of ‘Pending Approval’, ‘Override’ of ‘Approved’ or;
  • If any proposed order have been transitioned to a state of ‘Pending Approval’ or ‘Override’ and there are proposed orders present in a state of ‘Pending Rejection’ or ‘Rejected’;

All proposed orders will be transitioned to a state of ‘Exception’.

Note: This compliance rule will be an Entity level compliance rule, i.e. users must select ‘Entity’ in the ‘Apply Rule to’ field.

Function type

Process Template – Run Compliance Rules.

Note: Please refer to section 4.3.7.1 Run Compliance Rules for further details.

No tolerance required

 Check confirmation date is a valid trade date

Processing Phase

Approval Validations

Description

When cash allocation is run, each proposed order calculates an ‘Estimated Confirmation Date’ based on the ‘Start Date’ and ‘Trading Frequency’ attributes configured in the ‘Dealing Details – Subscription / Redemption’ tab under the Order Mgmt’ of the underlying asset that the entity is invested in.

When users have been granted the access to override the ‘Estimated Confirmation Date’ in the ‘Confirmation Date’ field on the proposed orders, the ‘Estimated Proceeds Date’ will also be updated accordingly.

Note: For details regarding override ‘Estimated Confirmation Date’, please refer to Section 5.2.2.2 Maintain Proposed (& Master) Orders.

This function will validate whether the ‘Confirmation Date’ is a valid trade date based on the configuration in the ‘Start Date’ and ‘Trading Frequency’ attributes.

  • If any proposed order has been transitioned to a state of ‘Pending Approval’ or ‘Override’ and the ‘Confirmation Date’ is not a valid trading date, then the proposed orders will be transitioned to a state of ‘Exception’. pControl will display an ‘Exception’ status with ‘Confirmation Date is not a valid business day’ message.

  • If any proposed order has been transitioned to a state of ‘Pending Approval’ or ‘Override’ and the ‘Confirmation Date’ is a valid trading date, then the proposed orders will be transitioned to a state of ‘Approved’. pControl will display an OK status with ‘Completed Compliance Rules’ message and the process will continue.

Note: This compliance rule will be a Proposed Order level compliance rule, i.e. users must select ‘Proposed Order’ in the ‘Apply Rule to’ field.

 Check if any external assets have missed cut-off

Processing Phase

Approval Validations

Description

This function checks if any external assets the entity may potentially invest in down the allocation structure have missed their cut-off time for the effective date of the process. The external entity attribute used for cut off time is referred to the ‘Notice Period Cut-Off time’ on Order Mgmt tab under ‘Dealing Details – Subscription / Redemption’.

Please refer to section 4.2.3 Specific Cash Allocation/ Rebalancing Attributes for further details.

When the proposed orders have been created, pControl will check the current time against the cut-off time configured in the ‘Notice Period Cut-Off time’ attribute

  • If the current processing time > notice period cut-off time on any external entities, the proposed order has missed the cut-off time. pControl will display an ‘Exception’ status with ‘Failed External Asset Cut-off Check’ message and the offending proposed order will have an ‘Exception’ state.

  • If the current processing time < notice period cut-off time on any external entities, the proposed orders have not missed the cut-off time. pControl will display an OK status with ‘Completed Compliance Rules’ message and process will continue.

Note: This compliance rule will be an entity level compliance rule, i.e. users must select ‘Entity’ in the ‘Apply Rule to’ field.

 Check Interfund Trading

Processing Phase

Approval Validations

Description

This function checks whether the asset on each proposed order, will ultimately invest into an external product that has been marked as being in a ‘special event’.

The positive liquidity allocation parameters are used to determine which type of assets the entity would invest into.

Special events include:

  • ‘Inactive’
  • ‘In liquidation’
  • ‘Closed to new investment’

If any of the above are true then the compliance rule will fail and an exception will be raised on the proposed order. The exception will need to be overridden to allow the cash allocation process to continue for the entity.

Function type

Process Template – Run Compliance Rules.

Insert link here to Run Compliance Rules dropdown

Entity data attributes used

If the entity will ultimately invest into external products with any of the following special events selected on the external product entities, the compliance rule will fail and an exception will be raised on the proposed orders.

  • Active – If not selected, this indicates the external product entity is inactive and therefore no trading will be allowed. The proposed order will be transitioned to ‘Exception’ state.

  • In Liquidation – If selected, the external product entity is considered to be in liquidation and therefore will not accept any purchase orders (transactions). The proposed order will be transitioned to ‘Exception’ state.

  • Closed to new investment – If selected, the external product entity will not accept any orders created from the new investors as there is no existing holding. The proposed order will be transitioned to ‘Exception’ state.

However the newly created orders will be accepted from investors where there is a pre-existing interfund relationship with the external products.

Note: The interfund trading compliance rules above can be applied either to Entity or Proposed Order, i.e. users can select either ‘Entity’ or ‘Proposed Order’ in the ‘Apply Rule to’ field.

Tolerance type

No tolerance required

Pre-requisite functions

Calculate proposed orders

Outputs

Proposed orders validated for special events.

 

 

 Check for Significant Cashflow

Processing Phase

Approval Validations

Description

This function ensures that users are made aware of any significant cashflows during the cash allocation process so they can determine if a dilution levy is applicable. The check is selected on the process template:

When selected, pControl will check if any proposed order amount of an asset as a percentage to the External NAV of that asset is greater than a specified percentage (Significant Cashflow Tolerance %), a warning will be issued with the message of ‘Breach Significant Cashflow Check for Asset XYZ’ in the processing status.

During manual approval of the proposed orders or when the proposed orders have been configured as ‘Auto Approved’ state, the proposed orders will be automatically transitioned to ‘Proposed Order Breach’ state indicating a significant cashflow breach and will appear on the ASM until the order is manually transitioned to one of the acknowledged states.

Notes: For further details regarding proposed order breach status stream, please refer to section 5.3.3 Proposed Order Breach Status Stream

The proposed order amount used in the calculation is the sum of proposed orders by asset (excluding Pending Rejected or Rejected states) i.e. All proposed orders in a state of ‘Pending Approval’ or ‘Override’.

The acknowledged states include:

  • Acknowledged With Dilution Levy
  • Acknowledged Without Dilution Levy

Once the proposed orders have been acknowledged, they will be removed from the ASM.

Note/s: This compliance rule is run as part of the Cash Allocation process but is primarily for the purpose of identifying significant cashflows for Unit Pricing purposes. ‘Ack without / with dilution levy’ is linked to the function [PRV120] – Check for Dilution Levy Acknowledgement in Unit Pricing process template.

The unit pricing process will raise exceptions if there are any significant cashflow proposed orders that have not been acknowledged.

This compliance rule utilises the value populated in the U070: External NAV buffer (recorded as an external balance). The validation is dependent on the loading of this value.

  • pControl Code in this buffer = the proposed order asset code
  • Effective Date = maximum effective date present for this asset <= the current effective date
  • External Nav = the Nav amount

Function type

Process Template

Entity data attributes used

Significant Cashflow Tolerance (%) – This attribute defines the tolerance level if exceeded will constitute a ‘Significant Cashflow’.

Tolerance type

 Tolerance defined in entity attributes see ‘Entity data attributes used’.

Pre-requisite functions

 U070: External NAV buffer where external NAV value of the assets of the proposed orders are loaded to pControl. Calculate Proposed Orders

Outputs

 Proposed orders identified for significant cashflow.

 

 Dual Authorisation Required for Amount/Unit Amendments

Processing Phase

Process End

Description

This function adds the dual authorisation (i.e. four-eyes) check to the unapproved proposed orders where the cash amount – ‘Prop Order (user)’ or the unit – ‘Unit Prop Order (user)’ field has been amended by the users and second person’s authorisation will be required prior transition the proposed orders to Approved state.

When this function is selected, pControl Rules will run and determine whether the proposed orders have been amended:

  • If the proposed orders have been amended – pControl Rules will transition these orders to Dual Authorisation and require a 2nd person to transition to the Approved state.
  • If the proposed orders have not been amended – pControl Rules will transition to the Approved state.

When this function is not selected on the processing template and users have amended the proposed orders, pControl Rules will not transition these amended proposed order to Dual Authorisation.

If the units in field ‘Unit Prop Order (user)’ have been amended, the ‘Is Unit Based’ field must also be selected, in order for the dual authorisation function to work.
