Cash Allocation Calculation

In the Processing Module

The 'Calculation' processing phase performs a number of calculations to generate a set of proposed orders for an entity.

The calculation processing template functions which are executed in the Calculations phase of cash allocation process are shown below.

Cash Allocation Functions

Calculate Starting Position / Weights

Processing Phase

Calculations

Description

This function calculates the starting positions for the entity in terms of the actual allocation weights into its investments and the amount it is currently holding in ‘liquidity’.

The Total Investment Value is first calculated and output to the status messages. This is calculated as

Total Investment Value = ‘Calculated liquidity balance’ + ‘Total Investments Holdings balance’.

Starting positions are then calculated as;

Investment Holdings value of an asset

Total Investment value

For example, If an entity holds the following investments:

Asset Holding Value

Asset 1

$100

Asset 2

$200

Liquidity

$50

 

The starting position for Asset 1 would be calculated as follows:

100/(100+200+50)

= 28.5714%

Similarly, Asset 2 would be calculated as 57.1429% and liquidity 14.2857%.

All starting positions are output to the status messages (in percent to 4dp).

 Calculate AU Value vs Liabilities Mismatch

Processing Phase

Calculations

Description

This feature applies to UK Life Companies that perform Asset/Liability Matching (ALM).

This function calculates the total mismatch value between AU Value and Liabilities.

The mismatch value =

Total Liabilities (COB + Intra-day + Manual Adjustment) – Total AU Value + Liquidity Accounts

When this function is selected, pControl will calculate the mismatch value between AU value and Liabilities including liquidity accounts as part of the proposed orders calculation.

Details of the calculation are displayed in the processing status message:

  • When the mismatch value is > 0, pControl will generate an Application proposed order
  • When the mismatch value is < 0. pControl will generate a Redemption proposed order

The example above showed a mismatch order of $650,000.00 (600,000 + 100,000 + 50,000 – 600,000 + 500,000). The AU internal Provider order is created with a value of $650,000.00.

At the end of the cash allocation process, pControl will also create an AU Registry order based on total value of the proposed orders that are in an Approved state. Any rejected proposed orders will not form part of the AU Internal Provider order. The example below shows two approved proposed orders ($30,000 each) and one rejected proposed order, pControl will only create the mismatch order of $60,000 (sum of the two approved proposed orders).

In addition, if tolerances are set on the ALPs of the investing funds / assets (i.e. ‘Minimum/Maximum/Rounding Amount’ or ‘Minimum/Maximum/Rounding Units’), pControl will adjust the value of the mismatch order (the AU Internal Provider orders) by these breaches in the proposed orders.

Function type

Process Template

Entity data attributes used

The AU value is calculated based on the units * price where:

  • Price = ‘Price Type for Valuations’ price type from the last approved price of the current entity on or before the current effective date.
  • Units = Registered Holders Total Units for the originator specified on the ‘Asset Unit Entity’ attribute.

Tolerance type

 None

  • Load Intra-day Liabilities
  • Load Manual Adjustment Liabilities

  • Last approved price of the current entity

 Calculate Proposed Orders

Processing Phase

Calculations

Description

This function calculates the proposed orders for the entity based on the information already loaded and calculated in pControl. The calculation is executed differently depending on the ‘Allocation Methodology’ selected on the entity which will be discussed in detail individually in the next section.

The available allocation methodologies in pControl are:

  • Straight Percentage
  • Zero Cash To Allocate
  • Towards Target
  • Rebalance to Target
  • Partial Rebalance to Percent Target
  • Box Management

Based on the allocation methodology, pControl will calculate the ‘Ideal Order’ either in cash or unit base for each investment and is then tested against the criteria set on the relevant ALP.

Prior to the creation of the proposed orders, a validation will be performed by pControl to ensure:

  • There is a latest approved price available AND
  • The price type configured on the fund should match with the price type on the asset’s latest approved price for any assets that the fund is investing in.

If there is no latest approved price, zero price or no matched price type available, an error status message with ‘No Latest Unit Price found for Asset X for price date DD-MM-YYYY for price type XYZ price. A price of 1.00 will be used for calculation of proposed order unit amounts for positive/negative orders’ will be displayed on the processing status.

The master proposed order with an Exception state will be displayed on the ASM. The cash allocation process can be transitioned to ‘Run Again’ or ‘Run in Error’.

The following tests are performed on the ‘Ideal Order’ once it has been converted to the currency of the asset (if this is different to the base currency of the fund):

Cash-Based Tests Description

Transaction Rounding Amount

If a rounding value has been configured on the ALP, the proposed order will be rounded to the nearest multiple of this value (applications are rounded down towards 0 and redemptions are rounded up towards 0). The resulting rounded order amount is written into the 'Rounded Order' field on the proposed order and if the order has been rounded, the rounded order flag will be selected on the proposed order.

Transaction Minimum Amount

If this has been set on the ALP, then the ‘Rounded Order’ will be tested against it and if the rounded order is less than the minimum amount, the ‘Proposed Order’ field will be set to 0 and the ‘Min/Max Breach’ flag will be selected on the proposed order record.

Transaction Maximum Amount

If this has been set on the ALP, then the ‘Rounded Order’ will be tested against it and if the rounded order is greater than the maximum amount, the ‘Proposed Order’ field will be set to the maximum amount and the ‘Min/Max Breach’ flag will be selected on the proposed order.

 

Unit-Based Tests Description

Transaction Units rounding

If a rounding unit value has been configured on the ALP, the proposed order will be rounded to the nearest multiple of this unit value (applications are rounded down towards 0 and redemptions are rounded up towards 0). The resulting rounded order is written into the ‘Rounded Order’ field on the proposed order and if the order has been rounded, the rounded order flag will be selected on the proposed order.

Transaction Minimum units

If this has been set on the ALP, then the ‘Rounded Order’ will be tested against it and if the rounded order is less than the minimum units, the ‘Proposed Order’ field will be set to 0 and the ‘Min/Max Breach’ flag will be selected on the proposed order record.

Transaction Maximum units

If this has been set on the ALP, then the ‘Rounded Order’ will be tested against it and if the rounded order is greater than the maximum units, the ‘Proposed Order’ field will be set to the maximum units and the ‘Min/Max Breach’ flag will be selected on the proposed order.
  • The ‘Proposed Order’ field on the Proposed Order is updated to the amount that is the result of these tests.
  • The ‘Transaction Type’ is set on the Proposed Order to be the:
  • ‘Application Transaction Type’ if the ‘Proposed Order’ amount is greater than 0
  • ‘Redemption Transaction Type’ if the ‘Proposed Order’ amount is less than 0
  • The appropriate flags will be selected on the proposed orders:
  • Is Unit Based? – the proposed orders are unit base orders
  • Min/Max Breach? – the proposed orders have breached min/max transaction amount/unit
  • Order Rounded? – the rounding tests have been applied to the proposed orders
  • Full Redemption? – the proposed orders have breached the ‘Cannot have Short Positions’ in entity attribute.
  • Cash Reallocated? – proposed orders have spare cash and ‘Reallocate Spare Cash’ has been selected in process template.

Note: All information on these calculations and tests is written out as status messages.

Function type

 Process Template

Entity data attributes used

The following attributes are used by all cash allocation methodologies (unless otherwise specified):

  • Exclude Unapproved Orders - if selected, orders which are ‘Unapproved’ will not be included in the ‘Cash to Allocate’ calculation i.e. where registry orders are not configured to be created in ‘Approved’ state and have not been manually transitioned to ‘Approved’.

Note: For further information on configuration required to create registry orders in an ‘Approved’ state, refer to section 4.2.5 Default Processing Attributes.

  • Rounding Method – This rounding method attribute is used in conjunction with the ‘Rounding amount’ or ‘Units rounding’ configured on the allocation parameters to determine how the proposed orders are created. The rounding methods are discussed below:
  • Round Down:

The negative amounts (typically redemption) are rounded to the nearest multiple of the rounding amount that is FURTHER from zero.

The positive amounts (typically application) are rounded to the nearest multiple of the rounding amount that is CLOSER to zero.

  • Round Nearest:

Both positive (typically application) and negative (typically redemption) amounts are rounded to the nearest multiple of the rounding amount.

  • Round Towards Zero:

Both positive (typically application) and negative (typically redemption) amounts are rounded to the nearest multiple of the rounding amount that is CLOSER to zero.

  • Round Using Rules: Will use client rounding rules
  • Spare Cash – This determines how the residual cash should be allocated. If no option is selected, pControl will default to ‘Largest Investment Holdings (ALP)’.

Note: For further information on spare cash, please refer to section 4.3.5.4 Reallocate Spare Cash.

  • Cannot Have Short Positions– Where a redemption transaction would generate a sale of an underlying asset which is greater than the position held i.e. generate a short position, the selection of this attribute will only allow a ‘full redemption’ transaction to be generated i.e. the full ‘confirmed’ holding will be redeemed and the order marked as a ‘full redemption’. The remaining redemption (which cannot be funded) is reflected as negative ‘Spare Cash’ balance to be funded through future cash allocation processing.
  • Proposed Orders
  • Application Transaction Type- This determines which transaction type will be used for the ‘Proposed Order’ where the calculated value is > 0.
  • Redemption Transaction Type- This determines which transaction type will be used for the ‘Proposed Order’ where the calculated value is < 0.
  • Application Price Type -The price type that will be used to calculate the unit value of applications for the proposed order into the underlying asset e.g. NAV Price.
  • Redemption Price Type – The price type that will be used to calculate the unit value of redemptions for the proposed order into the underlying asset e.g. NAV Price.
  • Application Dealing Basis – This determines whether cash or units will be used to process applications.
  • Redemption Dealing Basis – This determines whether cash or units will be used to process redemptions.

Tolerance type

 No tolerance required

Pre-requisite functions

The following functions are optional pre-requisites but will impact the proposed orders calculated by this function:

  • Load Investment Holdings
  • Calculate Liquidity Balance (Cash to Allocate)
  • Load Calculated and Approved Proposed Orders from Investors
  • Load all Orders and Calculated Proposed Orders

Description

The cash available to allocate (liquidity balance) is divided up amongst the investments specified by the Allocation Parameters (ALPs). This creates a proposed order for each investment (application or redemption). This value is written into the ‘Ideal Order’ field on the Proposed Order.

Note: Refer to the methodologies section in the Cash Allocation user guide for further information. 3.1.11

Zero Cash to Allocate

Processing Phase

Calculations - Additional Information

Description

This methodology allows users to have the option not to allocate cash to funds when a cash allocation and rebalancing process is run, even though target weights and liquidity have been set up in an ALPs. Proposed orders with zero value will be created, and this provides the opportunity to adjust the proposed order amounts, or to leave the values at zero. The ‘Zero Cash to Allocate’ method will simply zero out the ideal order value on the proposed orders, even when there is cash to allocate. The starting position, available cash, etc. remain viewable on the proposed order.

Note: Refer to the methodologies section in the Cash Allocation user guide for further information. 3.1.11

Entity data attributes used
  • Allocation Methodology = Zero Cash To Allocate
  • Allocation Grouping – if blank, pControl will default to ‘Effective Date’

Tolerance type

 No tolerance required

Pre-requisite functions

 None

Outputs

Proposed order with no value generated.

 

 

Towards Target

Processing Phase

Calculations - Additional Information

Description

This methodology will take the cash to allocate and propose orders that will move investments that are furthest away from their target back to target first. Money is given to overweight or underweight assets in proportion to the value of how overweight or underweight they are. Where the cash available to allocate (liquidity) balance is positive, the application order will propose to get the investment that is furthest away from target (underweight) back to target. If there is still cash left once all underweight investments are back to target (or if there are no underweight investments to begin with), then the remaining cash will be allocated according to target weights across all investments, in proportion to the under/overweight value. The same methodology is applied to negative cashflows and overweight investments.

Note: Refer to the methodologies section in the Cash Allocation user guide for further information. 3.1.11

Entity data attributes used
  • Allocation Methodology = Towards Target
  • Allocation Grouping – leave blank, pControl will default to ‘Effective Date’

Tolerance type

 No tolerance required

Pre-requisite functions

The following functions are optional pre-requisites but will impact the proposed orders calculated by this function:

  • Load Investment Holdings
  • Calculate Liquidity Balance (Cash to Allocate).
  • Load Calculated and Approved Proposed orders from Investors (for Box Management Methodology only)
  • Load All Orders and Calculated Proposed Orders

Outputs

Proposed Orders

 

 Rebalance to Target

Processing Phase

Calculations - Additional Information

Description

Rebalancing is usually performed to re-weight any investment holdings that are out of tolerance. In pControl, this process calculates and creates proposed orders in the same way as Cash Allocation.

For each Allocation Parameter (ALP), pControl checks whether the starting position in each asset specified by the ALP is outside of the tolerances that have been set, i.e. the following test is performed.

Insert equations

If Starting Position (Start Weight %) < (Target Weight – Lower Tolerance) OR Starting Position (Start Weight %) > (Target Weight + Upper Tolerance) then propose a rebalancing order because the investment is outside of tolerance.

If (Target Weight – Lower Tolerance) < = Starting Position (Start Weight %) < = (Target Weight + Upper Tolerance) then pControl will not propose a rebalancing order because the investment is within tolerance.

Note: A rebalance will be initiated if any of the starting positions are outside of the tolerance i.e. also applied for the liquidity entity. If there is an amount in cash when the rebalancing runs, and it is outside of the upper tolerance, then the rebalance will generate a proposed order to bring the holding to within the upper limit.

For each investment that has been identified as being outside of tolerance, an ‘Ideal Order’ is calculated as follows:

Insert equations

Ideal Order = (Target weight into Asset * Total investment value of the Fund) - Starting value of investment in the asset

Entity data attributes used
  • Allocation Methodology = Rebalance to Target
  • Allocation Grouping – leave blank, pControl will default to ‘Effective Date’

Tolerance type

 No tolerance required

Pre-requisite functions

The following functions are optional pre-requisites but will impact the proposed orders calculated by this function:

  • Load Investment Holdings
  • Calculate Liquidity Balance (Cash to Allocate).
  • Load Calculated and Approved Proposed orders from Investors (for Box Management Methodology only)
  • Load All Orders and Calculated Proposed Orders

Outputs

Proposed Orders

 

 Partial Rebalance to Percent Target

Processing Phase

Calculations - Additional Information

Description

The ‘Partial Rebalance to % Target’ methodology checks each Allocation Parameter (ALP) to determine whether the starting position into the asset specified by the ALP is outside of the tolerances that have been set.

If any investment is identified as being outside of tolerance, an ‘Ideal Order’ is calculated for all investments as follows:

Insert equation

Ideal Order = (Partial rebalance weight in asset * Total investment value of the fund) - Starting value of investment in the asset ($ value)

‘Partial Rebalance Weight’ in asset = Percent towards target * Target weight in asset + (1 - Percent towards target) * Starting position (% value).

This value is written into the ‘Ideal Order’; field on the Proposed Order.

 Reallocate Spare Cash

Processing Phase

Calculations

Description

The ‘Calculate Proposed Order’ function (as shown above) calculates the ‘Ideal Orders’ that should be made into investments and then performs a number of tests on these ideal orders to generate a set of proposed orders, e.g., minimum transaction test and rounding. If an ideal order is rounded or fails one of the tests, the actual proposed order may be different to the ideal and there will be some ‘spare cash’ left over. This spare cash can be a positive or negative figure.

For example:

  • If the ideal order was an application for $550,000 but the maximum order into this asset was set to $500,000, the proposed order would be $500,000 and there would be $50,000 spare cash left over that will not be ‘allocated’.
  • If the ideal order was a redemption for $47,000 but the minimum order into this asset was set to $50,000, the proposed order would be $0 and there would be $47,000 spare cash left over that will not be ‘allocated’
  • If the ideal order was an application for $47,000 and the rounding was set to $5,000, the proposed order would be $45,000 and there would be $2,000 spare cash left over that will not be ‘allocated’.

This function will attempt to ‘reallocate’ the spare cash to the fund’s investments to ensure that the fund is fully invested.

There are four methods in which spare cash may be allocated to its underlying proposed orders depending on the option selected in the attribute on the entity.

All of the information on this cycle is reported out through status messages. A sample is shown below.

Note: If the ‘Straight Percentage (with orders criteria)’ methodology has been used and more than one criteria ‘bucket’ has been processed, spare cash will be reported by critera and will try to be allocated by criteria group using the following process.

  • The new proposed order amount is then tested again using the minimum, maximum and rounding settings from the ALP.
  • If any spare cash can be re-allocated to this investment within the tests that are set, the amount that has been reallocated is reported and the Reallocated Cash flag is set on the Proposed Order
  • If the spare cash remaining is still not zero, the next allocation investment is retrieved and the remaining spare cash is added to the Proposed Order and the tests are performed again.
  • This cycle continues until all of the investments have been processed.
  • Any remaining spare cash at the end of this cycle is reported and will remain as liquidity or cash at hand for the fund until the next cash allocation process is executed.

 

Function type

 Process Template

Entity data attributes used

There are four methods in which spare cash may be allocated to its underlying proposed orders:

  • Allocation Priority – The ‘Priority’ field on the Allocation Parameters (ALP) is used, with the number 1 being the highest priority setting. pControl will retrieve the investment with the highest priority number and reallocate the spare cash to its proposed order to create a new proposed order amount.
  • Largest Investment Holding (absolute) – The spare cash will be reallocated to the largest investment holding, by value. For example, if the current investment holdings value is $100 and $95 respectively, then the spare cash will be allocated to the $100 investment holdings.
  • Largest Investment Holding (ALP) – The target weights are used to determine the priority with which the spare cash should be allocated. All spare cash will go to the investment with the largest target weight. For example, if there is $100 spare cash and there are 3 ALPs for 40%, 40 % and 20%, then the spare cash is allocated to the first highest priority investment, i.e. the one with 40% target weight. If there is more than one investment with the same target weight, the first allocation that is encountered during processing is used.
  • Move Closer to Target – The spare cash will be reallocated to the investment that is furthest away from its target weight (after the initial allocation of cash, not as per starting position). Any spare cash will be allocated up to the investment’s weight, at which point any remaining spare cash would go to the next investment that is now furthest away from its target, and so on.

Note: If a ‘Spare Cash Method’ is not selected, pControl will default to ‘Largest Investment Holdings (ALP)’.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

Proposed Order amended by any ‘Spare Cash’ values and spare cash invested.
