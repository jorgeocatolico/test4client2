Cash Allocation Validate Data

In the Processing Module

The data validations processing phase validates any data that has been loaded to ensure it is appropriate for cash allocation processing to continue. The 'Validate Data' processing template functions which are executed in the Data Validations phase of processing are described below.

Cash Allocation Functions

Check Allocation Parameters Sum to 100%

Processing Phase

Data Validation

Description

This function checks that the set of Allocations Parameters (ALPs) that have been loaded for an entity sum to 100%.

If the ALPs sum to 100%, pControl will output a status message ‘Allocation Parameters add to 100% for FUNDX’ and processing continues.

The Cash Allocation process will stop, with Error status messages produced and an exception raised on the master proposed order if any of the following is true:

  • There are no ALPs loaded for the fund.
  • There are Application, Redemption AND Standard Allocation Types loaded for the fund (there should either be Standard types OR Application and Redemption types).
  • The sum of Standard Allocation Type ALPs for the Fund does not sum to 100%.
  • The sum of Application Allocation Type ALPs for the Fund does not sum to 100%.
  • The sum of Redemption Allocation Type ALPs for the Fund does not sum to 100%.
  • There is an Application Allocation Type ALP with no corresponding Redemption Allocation Type ALP, i.e. an application allocation type ALP into an asset with no corresponding redemption allocation type ALP into the same asset.
  • There is no ALP into the associated Liquidity Entity.

Function type

Process template

Entity data attributes used

None

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Check ALP Pending Change

Processing Phase

Data Validation

Description

This function checks that the set of Allocations Parameters (ALPs) that have been loaded for an entity do not have any pending changes in the ASM.

If there is any pending change that is related to the ALPs, pControl will raise an exception with a status message ‘ALP for FUNDX  ASSETY YYYY-MM-DD has pending change’ and the process will stop.

An exception will also raise on the master proposed order, users can have the following options to transition the master proposed:

  • Run Again – To run the cash allocation process again with pending changes authorised by an alternate user.
  • Run in Error – To cancel the cash allocation process.

Function type

Process Template

Entity data attributes used

None

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Check all Holdings have an Allocation Parameter - NOT IN pCONTROL

Processing Phase

Data Validation

Description

This function checks that there is an Allocation Parameter (ALP) in pControl for each of the entity’s current set of investment holdings.

Note: This check can be used if an entity should only ever hold investments / assets as defined by the target investment structure (set of ALPs).

If the fund has an investment holding balance and there is;

  • An ALP for each investment holding balance (including the liquidity entity), the test passes and processing continues normally.
  • No ALP for any of the investment holdings, the cash allocation process stops, an error is raised in the processing status messages and an exception is raised on the master order.

Note: This is a mandatory check to ensure that the cash allocation process includes all investment holdings. Therefore, it should always be selected.

Function type

Process Template

Entity data attributes used

None

Tolerance type

No tolerance required

Pre-requisite functions

None

 Check  Units on Issue

Processing Phase

Data Validation

Description

This function checks the pControl internal units on issue against the units on issue in the U108: Units on Issue import buffer.

  • Where the pControl internal confirmed units are less than the unit threshold value, the validation will not be run as part of the process, the process will continue.
  • Where the pControl internal confirmed units exceed or equal the unit threshold value, the validation will run as part of the processing and
  • If data is present in the U108: Units on Issue buffer for the effective date and the unit difference is within the ‘Unit Difference Tolerance’ specified - the status messages will show an ‘OK’ status and processing will continue.
  • If data is present in the U108: Units on Issue buffer for the effective date, but the unit difference is outside the ‘Unit Difference Tolerance’ specified - the status messages will show an ‘Exception’ status along with the message ‘Unit Difference XXX exceeds UOI Check Unit Difference Tolerance of YYY’. The master proposed order status stream will transition to an ‘Exception (Override Allowed)’ and can be transitioned to ‘Override’, ‘Run in Error’ or ‘Run Again’. When transitioned to ‘Override’ the process will continue to the ‘Calculations’ phase.
  • If there is no data present in the U108: Units on Issue buffer for the effective date, the status messages will show an ‘Error’ status along with the message ‘Record is not present in u108.for unitholder XXX and asset YYY’. The master proposed order status stream will transition to an ‘Exception’ and can only be transitioned to ‘Run in Error’ or ‘Run Again’.

Note: For further information on the status streams, refer to Section 5.3

Function type

Process Template

Entity data attributes used

  • Unit Threshold – This validation will only be performed when pControl confirmed units are greater than this threshold value i.e. if pControl Units = 50,000 and unit threshold is 100,000 this validation will not be performed.
  • Unit Difference Tolerance – This is the tolerance established for the validation i.e. the validation will breach if the unit difference between the confirmed units and the U108 confirmed units is greater than this tolerance i.e. If pControl confirmed units = 50,000 and U108 confirmed units =50,100 and the unit difference tolerance = 100 the validation will pass. If the unit difference tolerance = 50 the validation will fail.

Tolerance type

Defined on entity attributes

Pre-requisite functions

Load Investment Holdings

Outputs

Validated UOI

 

 Validate COB AU Value vs Liabilities Mismatch

Processing Phase

Data Validation

Description

This feature applies to UK Life Companies that perform Asset/Liability Matching (ALM).

Note: For more information on Life Company processing, please refer to Appendix D – UK Life Company Processing.

This function checks whether the mismatch value (between COB Liabilities value and AU Value) is within the tolerance amount specified in the ‘Mismatch Tolerance Amount’ in the ‘AU Value vs Liability Mismatch’ attribute.

The COB AU Value vs Liabilities Mismatch = Total COB Liabilities – Total AU Value

  • When this function is not selected, no validation will be performed on the mismatch amount and processing will continue.
  • When this function is selected and the ‘Mismatch Tolerance Amount’ is populated:
  • If the mismatch amount is within the tolerance amount - the status messages will show an ‘OK’ status and processing will continue.
  • If the mismatch amount is outside the tolerance amount - the status messages will show an ‘Exception’ status along with the message ‘Difference between COB AU Value and Liabilities $ amount exceeds Mismatch Tolerance Value $ amount’. The master proposed order status stream will transition to an ‘Exception (Override Allowed)’ with the exception message displayed on the ‘Processing Note’ field and can be transitioned to ‘Override’, ‘Run in Error’ or ‘Run Again’. When transitioned to ‘Override’ the process will continue to the ‘Calculations’ phase.

The COB Liabilities amount is loaded from the U306 buffer either through:

  • Amount populated in ‘Value’ field or
  • Amount calculated from ‘Units’ field multiplies by the last approved price type specified in the U306 worktable.

The AU Value will be discussed in detail in the Calculation Configuration phase, please refer to Section 4.3.5.2 Calculate AU Value vs Liabilities Mismatch.

Function type

 Process Template

Entity data attributes used

The mismatch tolerance amount must be populated when this function is selected.

Tolerance type

 Defined on entity attributes

Pre-requisite functions

COB Liabilities amount from U306 worktable.

Load investment holdings from 'Asset Unity Entity' and calculate the AU Value.

Outputs

 Validate Mismatch Value
