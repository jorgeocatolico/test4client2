Cash Allocation Process Validation

In the Processing Module

The Process Validation phase initialises i.e. sets up the environment so that the processing can take place successfully and performs some checks to ensure that it is valid for the process to be running on a particular entity.

 

Cash Allocation Functions

Check if entity is set for cash allocation

 

Processing Phase

Process Validation

Description

This phase checks whether the entity is configured to be included in Cash Allocation processing.

  • If the entity is configured to run cash allocation, the status messages will output an ‘OK’ status along with the message ‘Entity XYZ is set to run cash allocation’ and continue processing.
  • If the entity is not configured to run cash allocation, the status messages will output an ‘Error’ status along with the message ‘Entity XYZ is not set to run cash allocation’ and processing will go into a wait state awaiting user action.

Note: The master proposed order status stream will also transition to ‘Exception’.

Function type

Process Template

Entity data attributes used

Include in Processing – this attribute determines whether the entity should be included in the processing run. This attribute is particularly useful when processing is requested on a group entity, (i.e. bulks submit to request processing for several entities). The bulk-submit process will check this attribute to determine whether to process all entities within the group, or only a selection of the entities.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Check if valid cash allocation day

Processing Phase

Process Validation

Description

This check determines if the effective date selected for the process is a valid ‘working day’ or not.

  • If the processing day is a working day, the status messages will output an ‘OK’ status along with the message ‘DD-MMM-YYYY is a valid cash allocation day’ and continue processing.
  • If the processing day is not a working day, the status messages will output an ‘Exception’ status along with the message ‘DD-MMM-YYYY is not a valid cash allocation day’ and processing will go into a wait state awaiting user action.

Note: The master proposed order status stream will also transition to ‘Exception’.

Function type

Process Template

Entity data attributes used

There are three choices available for supplying information to pControl on valid working days. These are explained below.

Calendar – determines the calendar within pControl which will be used to check whether the processing day is a valid day to run a Cash Allocation / Rebalancing process for the entity.

Use Alternate Calendar Source – allows calendar information to be brought into pControl from an external source (L390: Working Days worktable) rather than using pControl Calendars.

Calendar Schedule Entity – determines the calendar schedule entity within pControl which will be used to check whether the processing day is a valid day to run a Cash Allocation / Rebalancing process for the entity.

Calendar data is required for both the Cash Allocation and Rebalancing processes to allow them to run on different frequencies e.g. Cash Allocation on a straight percentage basis may run each day and then Rebalancing may occur once a month. The relevant calendar data for the process being executed will be used for the check.

With Ad-hoc Rebalancing, ‘Rebalancing Calendar Schedule Entity’ is used to govern when ad-hoc rebalancing should be allowed to run.

  • If the processing day is a valid rebalancing day according to the rebalancing calendar schedule entity, the rebalancing process will continue.
  • If the processing day is NOT a valid rebalancing day, the processing status message will display ‘dd-mm-yyyy is not a valid schedule rebalancing day’. The master proposed orders will be set to ‘Finish’ and the rebalancing process will end with no proposed orders created.

Note: Calendars are maintained in pControl Processing and System Administration mode.  Check if multiple Runs are allowed for a day

Processing Phase

Process Validation

Description

pControl records (as an external balance) the number of times the process has been run. If the ‘Check if Multiple Runs Allowed for a Day’ function is selected on the process template, pControl will validate this record i.e. is this external balance greater than 1 thus testing whether there has been a previous run of the process for the entity on the same effective date.

Function type

Process Template

Entity data attributes used
  • If there has not been more than one process run, processing will continue.
  • If there has been more than one process run, the process will then look to the process entity attribute ‘Allow Multiple Cash Allocation Runs on a day’

  • If the attribute is selected, the status messages will output an ‘OK’ status along with the message ‘Entity is allowed to run multiple times. Previous run id found with batch id X’ and continue processing.
  • If this attribute is not selected, the status messages will output an ‘Exception’ status along with the message ‘Entity doesn’t allow multiple runs, previous run id found with batch number X’ and the processing will go into a wait state awaiting users to action.

Note: The master proposed order status stream will also transition to ‘Exception’.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Check if multiple unapproved proposed orders are allowed for a day

Processing Phase

Process Validation

Description

This check tests if the entity allows more than one ‘Unapproved Proposed Order’ for a given effective date / originator / counterparty relationship.

Function type

Process Template

Entity data attributes used
  • If there is only one Unapproved Proposed Order for the given effective date / originator / counterparty relationship, processing will continue.
  • If there is more than one Unapproved Proposed Order for the given processing date, a check is made of the ‘Allow multiple Unapproved Proposed Orders’ attribute on the entity.

  • If the attribute is selected, the status messages will output an ‘OK’ status along with the message ‘Previous calculated / unapproved proposed orders are allowed’ and continue processing.
  • If this attribute is not selected, the status messages will also output an ‘OK’ status along with the message ‘Previous calculated / unapproved proposed orders are NOT allowed’ and pControl will then delete the previous calculated / unapproved proposed orders and output the following message ‘Delete Previous Unapproved Proposed Orders. Unapproved Proposed Orders deleted X’ (X being the count of orders deleted).

Note: The previous proposed orders will be deleted from the proposed order view.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Check if external assets have missed cut-off

Processing Phase

Process Validation

Description

This checks if any external assets this entity may potentially invest into have missed their cut-off for the effective date of the process.

  • If the effective date of the cash allocation process < current date, the status message will output an ‘Exception’ status along with the message ‘Missed dealing cut-off for one or more assets’ and the processing will go into a wait state awaiting users to action.
  • If the effective date of the cash allocation process = current date, the status messages will output an ‘OK’ status along with the message ‘All assets within cut-off’ and continue processing.

Function type

Process Template

Entity data attributes used

The external asset must contain the following:

  • P1_TRANSACTION_EXTERNAL attribute group
  • Notice Period (offset) value that indicates when the cut-off date is
  • Notice Period cut-off time that indicates the time at which cut-off occurs
  • Notice Period Calendar to be used in conjunction with the Notice Period.

Tolerance type

None

Pre-requisite functions

For the external cut-off to be used, there must be Allocation Parameters defined through to the external assets. There may be several levels of investment before the external assets.

Outputs

None
