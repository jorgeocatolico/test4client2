Cash Allocation Process Cashflows

In the Processing Module

For cash allocation processing to run, pControl requires incoming / outgoing cashflows i.e. application / redemption orders to be present for the top level (receiving) entities. These orders represent the transactions from transfer agencies and registries, and are therefore known as ‘registry orders’.

The creation of cashflows/registry orders is incorporated into the cash allocation process via the selection of new process template functions / attributes as shown below.

Cash Allocation Functions

Generate Cashflow Records from Unit Movements

Processing Phase

Process Cashflows

Description

This function is typically used by Life Companies for the generation of cashflow records from unit movements based on unit balances loaded into the U108: Units on Issue input worktable as sourced from Life Company policy administration systems.

This function compares the total units on issue loaded today against the total units on issue at the previous working day to calculate a unit inflow or outflow.

Note: The unit inflow / outflow calculated here are used in later functions to calculate a cash amount cashflow record.

Function type

Process Template

Entity data attributes used

Note: The following attributes are located on the policy administration system entity – they define the parameters of the originator of the entries.

Multiple policy administration systems can be used to originate these entries. The parameters below can differ depending on the particular policy administration system.

Calculate Cash Amount?

  • If this attribute is selected, pControl will calculate a cash amount and populate the ‘Cash Amount’ field together with the field ‘Calculate Cash Amount’ selected in the U110 Cashflows and Orders worktable.

The cash amount will be calculated based on the unit movements between the two working days, X and the latest available price.

  • If this attribute is not selected, pControl will calculate the unit movements and populate the 'Unit Amount' field.

  • Price Type - The type of price i.e. Buy Price, Sell Price etc that should be recorded against the record in the U110 worktable.
  • Positive Transaction Type - The transaction type that should be recorded against the record in the U110 worktable if the value is a positive amount (i.e. Application).
  • Negative Transaction Type - The transaction type that should be recorded against the record in the U110 worktable if the value is a negative amount (i.e. Redemption).

Tolerance type

No tolerance required

Pre-requisite data

In order for pControl to generate the cashflow records, the following additional data is required.

U108 Units on Issue – there should be two records present in the U108 worktable with the ‘Unit Mvt Status’ = ‘L’ (for loading) and ‘Calc Unit Mvt’ selected.

Note: In order to view ‘Unit Mvt Status’ and ‘Calc Unit Mvt’ fields, the U108 worktable must be in ‘Details’ view (i.e. the ‘Details’ box must be selected on the work area toolbar).

Outputs

U110: Cashflows and Orders records containing the unit amount (or with a zero unit amount in preparation for the ‘Calculate Cash Amounts on Cashflow Records’ function).

 

 Calculate Cash Amounts on Cashflow Records

Processing Phase

Process Cashflows

Description

This function is typically used by Life Companies to convert unit cashflow records created from the 'Generate Cashflow Records from Unit Movements' function into a cash amount, and then record them in the U110: Cashflows worktable.

pControl will retrieve the latest approved unit price for the entity, price run type and price date and will use this price to calculate the cash value of the inflow / outflow (to 2dp).

Note: The records in the U110 worktable now contain the cash value and will be used in the ‘Create Registry Order’.

Function type

Process Template

Entity data attributes used

None

Tolerance type

No tolerance required

Pre-requisite functions

None

Note: The previous ‘Generate Cashflows from Unit Movements’ function if selected has prepared the U110 worktable records for the calculation of cash. The previous function is not a pre-requisite function – if not selected, the U110 records can be populated directly with all requirements for this function to run.

Outputs

The record in the U110: Cashflows and Orders worktable is updated with the cash value.

 

 Create Registry Orders

Processing Phase

Process Cashflows

Description

This function is used to create registry orders using the data in the U110: Cashflows and Orders worktable. These orders are created via the Cash Allocation process based on the ‘Unit Amount’, ‘Cash Amount’ or the combination of both populated in the U110 worktable.

The registry orders generated simulate the cashflow into the top-level entities that may receive their cash from external registries or transfer agencies, or through direct investment from investors.

Note: This function may also include the creation of ‘Reservation Orders’ if these are included in the U110: Cashflows import worktable. If included, these may be subsequently reversed see section 4.3.3.1 Reverse Previous Reservation Orders.

This function incorporates the creation of registry orders into the Cash allocation process, replacing the need to run the general process ‘Create Registry Orders’.

Function type

Process Template

Entity data attributes used

Date to use for Cash Settlement – allows a choice to be made regarding the date that will be used for the cash settlement of registry orders i.e. the ‘Price Date’ or the ‘Effective Date’ of the U110: Cashflows and Orders record which will influence when the registry transaction hits the configured accounts.

Note: If left blank, ‘Price Date’ is the default selection.

Note: The following attribute is located on the Default Processing Attributes entity.

Create Registry Orders in Approved Status - Registry orders can be created in an ‘Unapproved’ or ‘Approved’ state. For further details please refer to Section 4.2.5 Default Processing Attributes.

Reservation Order Transaction Types - A reservation is the notification that cash is available to invest or redeem (i.e. for premium receipts, claim notifications and scheme transfers). For further information regarding Reservation Order transaction types refer to section 4.3.3.1 Reverse Previous Reservation Orders

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

Registry orders into the top-level or receiving funds (for registry cashflows and reservation orders).
