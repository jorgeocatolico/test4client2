LD Cash Allocation Data Load

In the Processing Module

The data load processing phase loads all of the data that is required by the cash allocation process from internal pControl balances and the standard import worktables. The data load processing template functions are explained further in the following sections.

 

Cash Allocation Functions

Reverse Previous Reservation Orders

 

Processing Phase

Data Loads

Description

A reservation is the notification that cash is available to invest or redeem (i.e. for premium receipts, claim notifications and scheme transfers). These transactions are not yet recorded in the PAS (Policy Admin System) or Registry, and, as a consequence, there will be no corresponding cash transactions produced by the PAS and advised to pControl for Cash Allocation processing.

The reversal of reservations is typically performed in UK Life Company environments. For more information on Life Company processing, please refer to Appendix D – UK Life Company Processing.

Reservation transactions can be included in the cash allocation process. These transactions can be loaded into pControl from the U110: Cashflows import worktable, and identified through the use of a ‘reservations’ transaction type. The reservations will be created as registry orders and the amount of the order included in the cash to allocate and is allocated to the entity’s assets according to the ALPs and allocation methodology specified.

Note: For the registry orders to be created, the ‘Create Registry Orders’ function must also be selected on the process template.

When ‘Reverse Previous Reservation Orders’ is selected, the prior day’s reservation transactions will be reversed by writing an order using the ‘reservation reversal transaction type’ that is indicated on the Company Default entity.

It is important that they are reversed out the next day as cash may have been processed through the PAS by then, and the transaction will be double counted if the reservation is not reversed.

Details of the reservation order reversals are output to the status messages of the ‘Data Load’ phase of processing.

The reversal order amount will be identical to the initial reservation order with a transaction type that has been configured in the Company Default entity. The reversal amount is included in the cash to allocate and allocated to the entity’s assets according to the ALPs and allocation methodology specified. However the ‘Price Date’ on the reservation reversal transaction will be equal to the ‘Trade Date / Confirmation Date’ field.

Function type

Process Template

Entity data attributes used

The transaction types used to identify reservation and reservation reversal must be set on the Company Default entity level on the Cash Allocation tab.

Reservation Order Transaction Types

  • Application Reservation – The transaction type used to reserve an application order (cash to be received) i.e. Reservation_Applications.
  • Redemption Reservation – The transaction type used to reserve a redemption order (cash to be paid out / redeemed) i.e. Reservation_Redemptions.
  • Application Reservation Reversal – The transaction type used to reverse an application reservation order (usually the reverse of the application reservation transaction type) i.e. Redemptions.
  • Redemption Reservation Reversal – The transaction type to reverse a redemption reservation order (usually the reverse of the redemption reservation transaction type) i.e. Applications.

Tolerance type

No tolerance required

Pre-requisite functions

Load of reservation transactions to the U110: Cashflows Import worktable.

Outputs

Reversal of prior day's reservation orders.

 Load Calculated and Approved Proposed Orders from Investors

Processing Phase

Data Loads

Description

If selected, this function ensures that the dependencies within an investment structure are managed in terms of making sure that all of the investors into an entity have had their cash processed before the fund itself starts its cash allocation process.

Status messages from a sample cash allocation process where this function is selected, and investor proposed orders are not calculated or approved is shown below.

Note: This function waits for investor proposed orders to be ‘calculated’ or ‘approved’ as the investor can be set to auto approve to a ‘calculated’ or ‘approved’ state. This allows processing to continue regardless of which of these is configured.

Using the below scenario, where Fund 1 is the counterparty and asset and Investor 1 and Investor 2 are the originators. If this function is selected on the process template for Fund 1, the process will stop and wait until Investor 1 and Investor 2 have Calculated, Approved (or Rejected) their orders into Fund 1. Only when orders are in these states will the Cash Allocation process continue and calculate the liquidity balance and starting positions for Fund 1.

Note: Investors 1 and 2 can reject the proposed orders into Fund 1. This evidences their completion of processing allowing processing to continue for Fund 1.

This function will also wait for Calculated, Approved (or Rejected) orders to be present where the entity is the counterparty only in an ALP. This is to support the Box Management methodology.

Note: For information regarding Box Management, please refer to section 3.1.11.6 Box Management.

Function type

Process Template

Entity data attributes used

The following attributes can be configured in the investors entities.

Auto Approve

  • Automatically Approve Proposed Orders – If this attribute is selected, pControl will automatically approve all proposed orders to a selective state. This attribute is to be used in conjunction with the ‘Automatically Approve Proposed Orders to State’ in Auto Approve Condition section, see below for further details.
  • Do Not Automatically on Warnings – If this attribute is selected, then pControl will not automatically approve proposed orders which have warning types configured in the ‘Issue Warning on these Events’ table in the Post-Calculation Validations tab. The list of warning types corresponds to the warning event columns displayed in the proposed order.

This attribute should be used in conjunction with the ‘Do Not Automatically Approve on Warnings’ in the Auto Approve Conditions section, see below for further details.

  • Automatically Approve Orders – This attribute determines whether orders written on approval of the proposed order should be automatically approved.

Auto Approve Conditions

  • Do Not Automatically Approve on Warnings – This attribute determines which type of order will not be approved on warning status, e.g. All proposed orders or a Single proposed order.
  • Automatically Approve Proposed Orders to State – This attribute is used in conjunction with ‘Automatically Approve Proposed Orders’. If populated, pControl will automatically approve the proposed orders to a selected state, i.e. ‘Approved’, ‘Calculated’.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 Load Balancing Options from Originator

Processing Phase

Data Loads

Description

This function is applicable for entities which are set up as ‘Default Options’ entities and is used in conjunction with the ‘Wait for Balancing Options’ function.

Note: Refer to Appendix E – Default/Balancing Options for further information on Default/Balancing Options.

When this function is selected, pControl will load the investment structure between the default option entities and their associated investment option entities from the L260: Cash Allocation - Default Options Mappings worktable.

If there is no data present in the L260: Cash Allocation - Default Options Mappings worktable, the cash allocation process will have an Exception status with ‘No Balancing Options found when expected’ message.

The L260 worktable will need to be populated, and the process ‘Run Again’ from the ‘Master Status’ of the master proposed order.

 Wait for Balancing Options

Offsetting has been further developed and is described in PR-4443

Processing Phase

Data Loads

Description

This function is applicable for entities which are set up as ‘Default Options’ entities and is used in conjunction with the ‘Load Balancing Options from Originator’ function.

Please refer to Appendix E – Default/Balancing Options for further information on Default/Balancing Options.

When this function is selected, pControl will look into the L260: Cash Allocation - Default Options static worktable of the default option entities then check whether the associated investment options have all completed their cash allocation process, i.e. all of their proposed orders for the processing day are ‘Approved’ or ‘Rejected’.

  • If there are no matching investment options found in the L260 worktable for the default option entity being processed, the cash allocation process continues as normal.
  • If there are matching investment options for the default entity in the L260 worktable, and any of the investment options have not completed their cash allocation process, the master proposed order will enter a wait state, waiting for their completion.

Once all of the Investment Option cash allocation processes have completed, the cash allocation process for the default option continues. The process will make all of the reversing transactions from the Investment Options (reversing the proposed orders ) using the ‘weight’ set in the static worktable, and will also make the transaction into the ‘Asset Class for Default Option Offset Trade’ entity if this attribute has been set.

If the ‘Asset Class for Default Option Offset Trade’ attribute has not been set, then the value of the reversing transaction becomes part of the liquidity of the Default Option when it does its own cash allocation process.

After completion of this function, the cash allocation process for the Default Option moves on and completes any other functions as dictated by the processing template configuration.

Function type

Process Template

Entity data attributes used

Offsetting Trades – There are two attributes related to the offsetting trades.

  • Check Offsetting Trades – If is selected for the default option entities, pControl will check that the offsetting trades for all default options do not have any rounding issues.

Note: Cash residual rounding tolerance is also used by this function. For details please refer to Section 4.2.5 Default Processing Attributes.

Retrieve and Approve Offsetting Trades to State – This attribute determines the trade state of the offsetting trades and the trade state of the proposed orders that will be offsetting.

  • Blank – pControl will display a Warning status with ‘No offset trade state has been set, defaulted to CALCULATED’ message and will behave the same way as the ‘Calculated’.
  • Approved – pControl will retrieve all proposed orders that are in ‘Approved’ state from the investment options and create the ‘Approved’ offsetting trades in the default options.
  • Calculated - pControl will retrieve all proposed orders that are in ‘Calculated’ state from the investment options and create the ‘Calculated’ offsetting trades in the default options.

Depending on client’s operational policy, the Investment Options could have either ‘Approved’ or ‘Calculated’ state configured in the ‘Approval’ phase of the cash allocation process.

  • Asset Class for Default Option Offset Trade – this attribute defines the entity that a Balancing (Default) Option will place all reversing transactions into during the cash allocation process.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

Reversing transactions created for all investment options linked to the balancing (default) option and potentially a transaction created into the offset asset class that has been set.

 

 Load COB/ Intra-day/ Manual Adjustment Liabilities

Processing Phase

Data Loads

Description

This feature applies to UK Life Companies that perform Asset/Liability Matching (ALM).

Note: For further information on Life Company processing, please refer to Appendix D – UK Life Company Processing.

Some Life Companies do not expect to generate a net new money cash flow for each charging series when performing their cash allocation process. Instead looking from a holistic view of a fund by comparing the total value of the liabilities against the total value of the assets, the difference is the mismatch that the Life Companies will make a dealing decision on.

pControl can calculate the total liabilities value by loading the U306: Cash Allocation Liabilities worktable. There are three different type of liabilities that can be loaded into pControl.

  • COB
  • Intra-day
  • Manual Adjustment

For each liabilities type, the ‘Units’ or the ‘Value’ can be entered in the worktable table to derive the total liabilities value.

  • If ‘Units’ field is used - Price type field should also be specified so that liabilities can be calculated.

Calculate Value = Units * Price, where price is the last approved price in pControl for the current entity on or before the current effective date for the price type specified in the ‘Price Type’ field.

  • If ‘Value’ field is used –‘Units’ field is not required to be populated.
  • For Manual Adjustments – only ‘Value’ filed should be populated.

Function type

Process Template

Entity data attributes used

When ‘Load COB Liabilities’ or ‘Load Intra-day Liabilities’ has been selected on the process template, and the above corresponding entity attribute is also selected:

  • If no data are present in the U306 worktable during cash allocation process, pControl will enter a wait state with a status message ‘XXX Liabilities Data not present in Buffer. Load is Mandatory. Entering Wait Mode’.
  • If there are data present in the U306 worktable, pControl will continue the cash allocation process.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Load Allocation Parameters - NOT IN pCONTROL

Processing Phase

Data Loads

Description

This function loads the Allocation Parameters (ALPs) for the entity.

Note: This option will usually be selected as the ALPs define an entity’s target investment structure and the target weights are used in the determination of the proposed orders. No proposed orders will be calculated if this function is not selected on the processing template.

Function type

Process Template

Entity data attributes used

For Cash Allocation Process:

‘Load Allocation Parameters (ALP) Mandatory’ attribute checks that ALPs are present for the entity when the Cash Allocation process is run.

When this attribute is selected and:

  • ALPs are present, processing will continue (ALP details will be loaded and reported in the status messages as shown below).

  • No ALPs present for this entity, the status messages will output an ‘Error’ status along with the message ‘No ALPs found’.

When this attribute is not selected and:

  • ALP is present in the entity, processing will continue (ALP details will be loaded and reported in the status messages).
  • No ALPs is present in the entity, processing will continue with warnings reported in the Data Load phase status messages.

The process will continue through to the Data Validations and Calculations phase. The process has validations on ALPs at this phase so it is likely that the process will stop at these validations.

ALP Type Positive Liquidity / ALP Type Negative Liquidity – Used to define the ALP types where there is a different ALP Type for Positive Liquidity (net applications) and Negative Liquidity (net redemptions).

For Rebalancing Process:

‘Allocation Parameter Type’ defines the ALP type when a Rebalancing process is run.

When Ad-Hoc rebalancing is required to run for the fund, ‘Ad-Hoc’ is selected in the ‘Allocation Parameter Type’ attribute. pControl will then check the presence of the Ad-Hoc ALPs.

  • When it is a valid rebalancing day and Ad-Hoc ALPs are present, Ad-Hoc rebalancing process will continue.
  • When it is a valid rebalancing day but no Ad-Hoc ALPs present, an exception will be raised in the processing status message ‘No ALPS found’.

When Nudge rebalancing is required to run and glide path ALPs are to be used for the fund, the following attributes must be populated:

  • Allocation Parameter Type = ‘Standard’ or “Ad-Hoc’ should be selected
  • Glide Path table must be populated:
  • ALPs from fund to underlying assets with an ALP start date equal to the Start Date on the glide path attributes table
  • ALPs from fund to underlying assets with an ALP start date equal to the End Date on the glide path attributes table
  • The effective date of the rebalancing run must be between the glide path start and end dates.

‘ALP Type Ad-Hoc’ this attribute should be set to any value (not Blank). However, if it is set to ‘Ad-Hoc’ and Ad-Hoc ALPs exist for the effective date, then the Ad-Hoc ALPs will be used instead of the glide path ALPs.

‘Glide Path’ table defines the date range and method that is used for nudge rebalancing.

  • Start Date – The start date of the glide path period.
  • End Date – The last day of the glide path period.
  • Method - Cash Flow and Rebalancing method.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

None

 

 Load Investment Holdings

Processing Phase

Data Loads

Description

This function loads all of the investment holding balances for the entity from pControl (i.e. from internal pControl investment holdings). The investment holding balances are loaded to support the calculation of starting positions for the entity’s investments.

The value of an entity’s investment holdings is calculated using:

  • ‘Total Units Held’ or ‘Confirmed Units’ (see entity data attributes used)
  • The latest available price of the underlying investments, (see entity data attributes used) using the ‘Price Type’ to value Investment Holdings’ attribute.
  • The price type and price date used are also provided on the proposed order produced by the cash allocation process.

Note: If the underlying investments are external products, pControl will support the dealing term price type configured in the dealing term entity. For further details regarding Dealing Term Entity, please refer to pControl User Guide – Initialisation.

  • Investment holding balances are converted into the base currency of the entity if the currency of the asset is different from entity’s base currency.

Note: If the value of the investment holdings is zero a Warning message will be created i.e. ‘Total Investment holdings is 0’.

Function type

Process Template

Entity data attributes used

The value of investment holdings is calculated using the following attributes:

Value Pending Units

  • Where this attribute is selected - the investment holdings will be valued using ‘Total Units held’ i.e. confirmed and unconfirmed units.
  • Where this attribute is not selected - the investment holdings will be valued using only ‘Confirmed Units’.

Price Type to Value Investment Holdings

  • This attribute contains a list of price types. The investments will be valued using the latest available prices as per the price type attribute selection.

Note: If the price type to value Investment Holdings does not match with the price type of the investment asset, an Error status message with ‘No Latest Unit Price found for Asset X for price date DD-MM-YYYY for price type XYZ. A price of 1.00 will be used for calculation of proposed order unit amounts for positive/negative orders’ will be displayed on the processing status.

The master proposed order with an Exception state will be displayed on the ASM. The cash allocation process can be transitioned to ‘Run Again’ or ‘Run in Error’.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

Total Investment Holdings value is calculated and displayed in the processing status message.

 Load Liquidity Account Balances

Processing Phase

Data Loads

Description

This function will load the liquidity account balances for use in the later calculations (calculate cash to allocate).

Function type

Process Template

Entity data attributes used

Liquidity Account Balances – This attribute references a Liquidity Account entity that defines which internal balances are to be retrieved and summed to generate a liquidity balance for the entity.

The liquidity entity listed in the above attribute stores the list of internal balances used in the liquidity calculation.

If the above ‘Liquidity Accounts’ entity is used by this function, the internal balances from each of the cash accounts listed above will be loaded and summed as follows, to generate the liquidity balance.

  • Balance of the CASH AT BANK as at the Processing Day (T) PLUS
  • Balance of the PURCHASES PAYABLE as at T+1 PLUS
  • Balance of the APPLICATIONS RECEIVABLE as at T+2 PLUS
  • Balance of the REDEMPTIONS PAYABLE as at T+3

The status messages will output ‘Info’ regarding the balances loaded.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

Calculated liquidity balance (at T and using Days Forward) for the liquidity entity associated with the fund on which cash allocation is being run.

 Load Liquidity Adjustments

Processing Phase

Data Loads

Description

This function loads liquidity adjustments entered (manually or automatically) into the U305: Cash Allocation - Liquidity Adjustments import worktable. These liquidity adjustments are used to adjust the liquidity balance calculation for an entity on a given date or date range.

Function type

Process Template

Entity data attributes used

Load Liquidity Adjustments Mandatory attribute - If selected, the liquidity adjustments must be present in pControl when the Cash Allocation process is run.

When this attribute is selected and

  • there are records present in the U305 worktable, these are loaded as part of the liquidity calculations function, and processing continues. pControl will output the details of the liquidity adjustments to the status messages.

In the example below, the liquidity balance of TRN_PROD1 will be decreased by $9,000 for all processing dates between 1st May 2014 and 30th May 2014.

Note: Care must be taken in the maintenance of these liquidity adjustments i.e. that the items are removed as the ‘actual’ cashflows occur.

  • There are no records in the U305 worktable for the processing day, processing will enter a wait state with the status message of ‘No Liquidity Adjustment Found – Entering Wait Mode’. pControl will wait for U305 records to be loaded into the worktable before continuing.

Note: Once the adjustments have been added to the buffer – the process can be restarted using the P183: Data Load Monitor.

Tolerance type

No tolerance required

Pre-requisite functions

None

Outputs

An adjusted liquidity value
