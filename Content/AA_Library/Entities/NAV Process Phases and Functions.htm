Versioning

[VA050] - Inherit Interprocess Overrides

 

Business Description

pControl will inherit any matching overrides from the last version (Run Id) of the nominated price run type.

Other processes can also be closed by this function. abc

Processing Detail

pControl will create all exception events for the current version being processed, and will then look back to the latest version of the price run type specified in the attribute below. For any exception event which has exactly the same description and was overridden in the previous version, pControl will carry forward that override to the current version including the comment and overridden by attributes.

The comment on inherited override events will be prefixed with ‘Auto Override’ to distinguish them and exceptions that inherited an override from the previous version.

If the 'Request Process to Close' check box is selected, pControl will look for any open processes for the ‘Price Run Type’ defined against ‘Inherit Inter-Process Overrides’ attribute. If there are any, pControl will close them as part of the inheritance function. I.e. In the example below, these attributes are defined for Daily Pricing'. When a Daily Pricing process is run, pControl will check for any incomplete 'Pre NAV' processes, and close these as part of the Daily Pricing process.

Note: Although this function is part of the 'Versioning' phase, the entity attributes appear on the 'Approval' tab of the fund, and the actual closure of the incomplete process will occur following the 'Data Load' phase of the Daily Pricing process.

[VC010] - Check if Entity is Set to Process

 

Business Description

This check tests whether a fund is configured to be included in processing.

This check is useful where 'bulk processing' is used to run a process for several funds simultaneously, as a way to include or exclude certain funds from that process.

Processing Detail

If the 'Include in Processing' check box is:

  • Selected, the check passes, and processing continues.
  • Not selected, the check fails and processing stops.

If the 'Exclude from Bulk Submit and STP Processing' check box is selected for a specified process type, that process type will not be initiated when Straight-through Processing (STP) or Bulk Submit processes are initiated. The only way to start this process type, will be manually, via Straight-through Processing (STP).

Entity Data Attributes Used
  • Include in Processing
    • If selected, the check passes, and processing continues - this includes processes submitted manually, or automatically via Straight-through Processing (STP) (STP) or Bulk Submit
  • Exclude from Bulk Submit
    • If check box is selected, this process will not be run when any Straight through Processing (STP) or Bulk Submit processing is run. Selecting this function, is a way to exclude a single process, when this fund belongs to a group, for which bulk processing is being run.

[HLD060] and [HLD060-PUI] - Money Market Holding Amortized Cost vs Market Value

 

Business Description

The [HLD060] and [HLD060-PUI] checks assist in identifying when a specific security contributes to the deviation of a fund's shadowThe term shadow refers to the per share market value of a money market fund and is expected to closely track or 'shadow' the per share value of the fund calculated using the amortized cost method NAV from the expected $1 NAV. [HLD060] is the bottom up check for [POV045] Money Market Amortized Cost NAV vs Shadow (Variable) NAV Check, which identifies when a fund is out of the acceptable range.

The [HLD060] tests if the difference between the amortised cost for the holding and the market value of the holding divided by the fund's NAV, exceeds a tolerance.

[HLD060-PUI], tests the difference between the amortised cost and the market value of the holding divided by the fund's shares outstanding.

Processing Detail

This validation check is performed on the value of the Instrument Group types or categories selected on the HLD060 or HLD060-PUI validation entity, as shown below.

Validation Formula (will fail if):

For the Categories or Instrument types selected, the [HLD060] check will fail if:

$\small Abs\ \normalsize \left [ \frac{ Amortized\ Cost\ +\ Market\ Value }{Fund's\ NAV} \right ] \small > \ Tolerance$

For the categories or instruments types selected, the [HLD060-PUI] check will fail if:

$\small Abs\ \normalsize \left [ \frac{ Amortized\ Cost\ +\ Market\ Value }{Fund's\ Shares\ Outstanding} \right ] \small > \ Tolerance$

[PRV025] - Check Significant Movement in Market

 

Business Description

This function checks for significant movement in the market, by checking the movement of the fund's key benchmark in basis points against a tolerance.

Processing Detail

This function is usually used for funds that are historically priced although it can be executed for forward priced funds.

The movement in the market is calculated by looking at the movement in the ‘Key Benchmark’ from the prior pricing point or previous effective date. The calculated movement is then compared against the tolerance.

$\small Movement\ in\ key\ benchmark\ =\ \left ( \frac{current\ benchmark\ value}{prior\ benchmark\ value} \right )\ -1$

Note: This validation check relies on validated benchmark values for the current effective date. Therefore there is a prerequisite function of either Valuation 1, or similar process to load and validate benchmark values.

Entity Data Attributes Used
  • Validation Benchmarks
    • Benchmarks associated with a fund are configured under the following attribute. This table may contain several benchmarks, however only one can be designated as the 'Key Benchmark'.
    AttributeDescription

    Benchmark

    The benchmark who's value will be used in the calculation.

    Note: The benchmark/s listed here must be configured as a benchmark entity within pControl, and the benchmark's value must be available at the time this check is run.

    Comparison Period

    The comparison period to be used for the validation (i.e. Prior Pricing Point, Month to date etc.)

    L200 Validation Filter

    If there are multiple tolerances defined in the L200: Tolerances table for a particular validation, they can be assigned to a particular benchmark in the table by utilising the L200 Validation Filter.

    E.g. If a validation check uses different benchmarks for domestic equity funds compared to international equity funds, different tolerances can be configured for these benchmark types as shown below. The same text used in the Validation Filter below is then used in the validation filter above.

    Note: This example uses POV010 validation check, however other validation codes can be selected in the first column.

    Fixed Adj. (bps)

    The Benchmark value for the comparison period may be adjusted by the Fixed Adjustment basis point amount specified here using the following formula:

    $\small Adjusted\ Benchmark\ =\ Benchmark\ Value\ (bp)\ +\ Fixed Adj.$

    Fee / Tax Adj

    The Benchmark value for the comparison period can be adjusted through the inclusion of Fees and/or Tax calculated in the pricing process. Options include:

    • Net of Fees
      • $\small Fee\ Rate\ (Actual\ daily\ fees\ rate)\ =\ \frac{\frac{\left (\Large Total\ Fees\ Provision\ +\ Total\ Performance\ Fees\ Provision\ \right )} {\Large GAV}} {\normalsize Days\ Since\ last\ price}$
       
      $\small Fee\ Adjustment\ Rate(bp)\ =\ Fee\ Rate\ \times Days\ in\ Period\ \times 10,000$
    • Net of Tax
      • Where pControl’s Tax Expense module is used to calculate tax, the effective rate calculated by pControl, and stored as an external balance is used in the calculation. If not, the tax rate as specified on the entity ‘Calculate Tax provision’ attribute (/100) will be used for the calculation.

      $\small Tax\ Adjustment\ Rate(bps)\ =\ Tax\ Rate\ \times 10,000$
     
    • Net of Fees and Tax
      • The benchmark is adjusted by the above Tax Rate first, and then by the above fee rate.
      Where pControl’s Tax Expense module is used to calculate tax, the effective rate calculated by pControl, and stored as an external balance is used in the calculation. If not, the tax rate as specified on the entity ‘Calculate Tax provision’ attribute (/100) will be used for the calculation.

      $\small Fee\ and\ Tax\ Adjustment\ Rate\ =\ \large \frac{\left ( Total\ Benchmark\ Movement \times Tax\ Rate \right )+\ \left (1\ -\ Tax\ Rate\ \times Fee\ Rate \right )} {Total\ Benchmark\ Movement\ \times 10,000}$

      $\small Total\ (adjusted)\ Benchmark\ Movement\ =\ Unadjusted\ Key\ Benchmark\ Movement\ - Fee\ and\ Tax\ Adjustment\ Rate$

      $\small Recalculate\ using\ days\ in\ period\ (based\ on\ the\ ‘Comparison\ period’\ column\ specified\ above)$

      $\small Validation\ Benchmark\ =\ Total\ (adjusted)\ benchmark\ movement\ \times \frac{\large Tax\ Rate\ +\ ((1\ -\ Tax\ Rate) \times Fee\ Rate\ \times Days\ in\ Period)} {\large Total\ benchmark\ movement\ \times 10,000}$
    • No Adjustment / Blank:
      • Benchmark is not adjusted

    Key Benchmark

[LC010] - Calculate Fees Provision

 

Business Description

This function calculates any fee provisions that need to be applied to the fund in the calculation of its 'Net Asset Value'.

Processing Detail

For each fee that is to be processed, as defined in the fee details table attribute, will occur depending on the fee type as defined on the fee entity.

Note: Where Load Bid / Offer Valuations has been selected and GAV’s have been calculated for each basis, all three GAV values will be used in the calculation of fees and subsequently three fee provision values will be calculated (Total Fees, Total Fees (Buy Price) and Total Fees (Sell Price)).

Entity Data Attributes Used
  • Policy for accruing non-working days

    • Before non-working period

    3 days of fees will be accrued on a Friday. Monday will have an accrual of 1 day of fees.

    After non-working period

    3 days of fees will be accrued on a Monday. Friday will have an accrual of 1 day of fees.

    Split evenly before and after

    2 days of fees will be accrued on a Friday, and 2 on a Monday, i.e. the non-working days, Saturday and Sunday are split evenly between Friday and Monday.
  • Accrue to month end only
    • Fees are accrued to the end of the calendar month, if it is a non-working period.
    If this attribute is set and the pricing day is immediately before or after a non-working period containing the last calendar day of the month, the policy for accruing non-working days is ignored and pControl will accrue the relevant number of days to accrue to the end of the month before the non-working period and from the start of the month after the non-working period. This function is often used to ensure that hard close, technical or month end performance pricing includes the relevant fee accrual to the end of the calendar month.
  • Fee Details

    • Attribute

    Fee Type

    The fee entity in pControl which represents the fee to be applied for this fund's process.

Multi Leg Derivatives

There are many types of financial instruments that can be classified as multi-leg derivatives, the majority of which can be classified as either forwards or swaps.

Forwards

A forward contract is a non-standard contract between two parties to buy or sell an asset at a specified price on a future date. A forward contract can be used for hedging or speculation. The party agreeing to buy the underlying asset in the future assumes a long position, and the party agreeing to sell the asset in the future assumes a short position. The price agreed upon is called the delivery price, which is equal to the forward price at the time the contract is entered into. Types of forwards include:

  • FX Forward
  • NDFs
  • Forward Rate Agreements

Swaps

Swaps are instruments whereby two parties exchange a series of cash flows over a period of time. Examples of swaps include:

  • Interest Rate Swap Example
  • Total Return Swap
  • Currency Swap
  • Equity Swap
  • Volatility Swap
  • Variance Swap
  • Commodity Swap
  • Dividend Swap
  • Asset Swap
  • Inflation Linked Swap

Note: There is a difference between multi-leg derivatives and multi-leg transactions. Multi-leg transactions typically form part of an overall trade strategy which contains more than one leg associated with a given order. Strips, Packs, Bundles and Calendar Spreads are examples of multi-leg transactions. In pControl, only multi-leg derivatives are catered for at this time.

 

 [MLH057] Multi Leg Holding Validation - FX Forward Example Calculation

To calculate an unrealised gain on a position, the formula is:

$\small Market\ Value\ -\ Cost\ =\ Unrealised\ Gain/Loss$

As an FX Forward has no cost, the formula is simplified to:

$\small Market\ Value\ =\ Unrealised\ Gain/Loss$

The below table represents data available in the U100: Investment Holdings worktable for an FX Forward. In this example, the fund manager has agreed a forward contract EUR vs USD to sell 1,000,000EUR for USD at a contractual price of 1.35 USD per EUR to settle in 6 months. At NAV date, the current EUR for USD forward rate to settle in 6 months is 1.3965 (the USD-EUR rate = 0.7160736).

The U100: Investment Holdings worktable holds two records that represent that FX forward position. One asset, or leg, with a corresponding long position of 1,350,000USD and another asset with a short position of -1,000,000EUR.

With a USD base currency at NAV date, the formula to calculate the Base Market Value of FX Forward is:

$\small Base\ Unrealised\ Gain/Loss\ = Trade\ Currency\ Quantity\ \times \left ( Contractual\ Forward\ Rate\ -\ Current\ Forward\ Rate \right )$

$\small Base\ Unrealised\ Gain/Loss\ =1,000,000\ \times \left ( 1.34\ -\ 1.3965 \right ) \\ \small Base\ Unrealised\ Gain/Loss\ =\ -46,504$

Therefore, the Base Market Value of FX Forward = -46,504USD

Business Description

This function validates the movement in the NAV price against the movement in benchmarks over the given time periods.

Processing Detail

This function validates the movement in the price type specified below against the movement in the benchmark over a given period. The difference, as calculated by the following formula, is compared to the L200: Tolerances worktable, and an exception raised where the difference exceeds the tolerance.

  • Prior pricing point calculations:
    • $\small Movement\ in\ Price\ since\ Prior\ Pricing\ Point - Movement\ in\ Benchmark\ since\ Prior\ Pricing\ Point$
    $\small where\ Movement = \normalsize \left ( \frac{Current\ Value}{Prior\ Pricing\ Point\ Value} \right )$
  • Month to Date calculations:
    • $\small Movement\ in\ Price\ MTD - Movement\ in\ Benchmark\ MTD$
    $\small where\ Movement = \normalsize \left ( \frac{Current\ Value}{Value\ on\ last\ working\ day\ of\ Prior\ Month} \right )\ \small-1$
  • Financial Year to Date calculations:
    • $\small Movement\ in\ Price\ FYTD - Movement\ in\ Benchmark\ FYTD$
    $\small where\ Movement = \normalsize \left ( \frac{Current\ Value}{Value\ on\ last\ working\ day\ of\ Prior\ Financial\ Year} \right )\ \small -1$

Swing Pricing

$\small Swing\ Adjusted\ Price\ Movement\ = Price\ Movement\ -\ Swing\ Effect$

 

$\small Current\ Pricing\ Basis\ \neq Previous\ Pricing\ Basis\, then:$

  • If Current Pricing Basis = 'Expanding' and Previous Pricing Basis = 'Contracting', then:
    • $\small Swing\ Effect\ =\ \normalsize \frac{\left ( Current\ Expanding\ Price\ -\ Current\ Contracting\ Price \right )} {Previous\ Price}$
  • If Current Pricing Basis = 'Contracting' and Previous Pricing Basis = 'Expanding', then:
    • $\small Swing\ Effect\ =\ \normalsize \frac{\left ( Current\ Contracting\ Price\ -\ Current\ Expanding\ Price \right )} {Previous\ Price}$

This check will test the value in the 'unexplained' category of the Account Movement Attribution report, and compare it to a tolerance. If the unexplained value exceeds the tolerance, an exception will be raised.

Processing Detail

This check will raise an exception when the 'Unexplained value' is greater than the L200: Tolerance value. Unexplained Values represent all constituent accounts that have not been assigned to an account-based category in the AMAS.

Unexplained Value Example

Prior Day

The Unexplained Values, or UNEXP_AMAS as it is referred to on the AMAS view, on the prior day AMAS was $140.00. The $140.00 includes $100.00 in AUD_INTEREST and $40.00 from BROKERAGE_FEES. Both accounts constitute part of the NAV and both accounts were not linked to an AMAS category on the prior day.

Current Day

AUD_INTEREST is mapped on the current day to account group BANK_INTEREST and has been mapped to the category code INT_AMAS. BROKERAGE_FEES is mapped on the current day to the account group MISC_FEES which maps to category code OTH_FEES_AMAS.

Unexplained Values

The Unexplained Values for the Value Previous on the AMAS (i.e. the prior day) will be 0.00. The Unexplained Values for the Value Current on the AMAS (i.e. the current day) will be Fund NAV - Total of constituent accounts.

 

Validation Formula (will fail if)

$\small Abs\left ( Unexplained\ Value\ \right ) >\ L200\ Tolerance\\ \small where\ Unexplained\ Value\ =\ Fund\ NAV\ -\ \small\sum\ constituent\ NAV\ accounts$

Entity Data Attributes Used
  • Calculate Unexplained Values
    • If selected, these values are calculated using the following formula

    $\small Unexplained\ Values\ =\ \small Fund\ NAV\ -\ \sum\ constituent\ NAV\ accounts$

Interdependencies
  • nil

Prerequisite Functions
  • nil

Outputs

 
[CAT014] - Unexplained NAV Impact

 

Business Description

This check relates to the Account Movement Attribution report and tests the value in the 'unexplained' category divided by Total NAV, and compares it to a tolerance. If the resulting value exceeds the tolerance, an exception is raised.

Processing Detail

This check will raise an exception when the calculated value is greater than the L200: Tolerance value.

Unexplained Value Example

Prior Day

The Unexplained Values, or UNEXP_AMAS as it is referred to on the AMAS view, on the prior day AMAS was $140.00. The $140.00 includes $100.00 in AUD_INTEREST and $40.00 from BROKERAGE_FEES. Both accounts constitute part of the NAV and both accounts were not linked to an AMAS category on the prior day.

Current Day

AUD_INTEREST is mapped on the current day to account group BANK_INTEREST and has been mapped to the category code INT_AMAS. BROKERAGE_FEES is mapped on the current day to the account group MISC_FEES which maps to category code OTH_FEES_AMAS.

Unexplained Values

The Unexplained Values for the Value Previous on the AMAS (i.e. the prior day) will be 0.00. The Unexplained Values for the Value Current on the AMAS (i.e. the current day) will be Fund NAV - Total of constituent accounts.

 

Validation Formula (will fail if)

$\small Abs\left ( \frac{ \large Unexplained\ Value}{NAV} \right ) >\ L200\ Tolerance\\ \small where\ Unexplained\ Value\ =\ Fund\ NAV\ -\ \small\sum\ \normalsize consituent\ NAV\ accounts$

Entity Data Attributes Used
  • Calculate Unexplained Values
    • If selected, these values are calculated using the following formula

    $\small Unexplained\ Values\ =\ \small Fund\ NAV\ -\ \sum\ constituent\ NAV\ accounts$

Interdependencies

[CAT016] - Balance Sheet NAV v Capital and Income NAV

 

Business Description

This validation check compares two categories on the Account Movement Attribution view:

  • the UPM_AMAS, the adjusted NAV from the Balance Sheet and
  • UPMI_AMAS, the adjusted NAV from the Income Statement.

Processing Detail

Where the difference between the two is greater than the L200: Tolerance value, an exception will be raised.

Validation Formula (will fail if)

$\small \left ( UPM\_AMAS\ -\ UPMI\_AMAS\ \right ) >\ L200\ tolerance$

[AP080] - Fund Group Review PR-6411 TC to review

 

Business Description

If selected, approving and releasing the primary entity's pricing process, will also approve and release any related fund entity's pricing process.

Processing Detail

Consolidated entities, Product entities and Pool entities can be identified as the primary entity, where they are linked to other entities, using the attributes shown below.

The processes to be approved within the fund group, must all have the same effective date, and price run type. (They may have different run ids)

Entity Data Attributes Used

The following attribute is used to identify entities as part of a group. The 'primary entity' of the group will be displayed in the 'Linked Pool' field as shown below. Each fund entity can be linked to a maximum of one primary entity.

  • [AP080] Fund Group Review

    • Fund Group Review
    If selected, this fund entity is the 'primary entity'.
    All NAV processes run for this fund and all funds of this group will be part of the fund group review, and be approved and /or released together.

    Wait for group to be approved
    This attribute only exists on the primary fund entity. If selected, the entire group is to be approved and released at the same time. If there are exceptions on any related funds, both the primary fund and all related fund's price records will be transitioned to 'Wait for Group Review' state, and will need to be manually reviewed.

    If not selected any related funds without exceptions will be transitioned to approved and released at the same time as the primary entity. If any of the related funds have exceptions, at the time the primary fund is approved, those related funds will not be approved, and will need to be reviewed individually after their exceptions are cleared. They will then require manual approval and transition to released.
