Validation Entities

Types of Validation entities

There are two types of validation entities in pControl: Asset Validation and Per Unit Impact Validation.

Asset Validation entities are used for validations which:

  • have a denominator which is included in the validation calculation i.e. absolute value checks where the denominator is 1.
  • require a basis point impact, calculated using a denominator of NAV/GAV.

    • Note: The 'NAV/GAV denominator' is a mandatory attribute which is specified on the entity the validation is being run for (Data Exports tab).

     

Per Unit Impact Validation entities are used for validations which:

  • require a denominator of units on issue.

    • Note: The units on issue denominator values can be sourced from within pControl or from an external source.

 

 Validation Entity Example Validation Entity attributes

The 'common' data attributes on the validation entity provide general information about the nature of the validation; the classification of the validation, the business description of the test being conducted and the actual formula that is used in the validation, i.e. the 'will fail if' scenarios

Attribute

Description

Common

Validation Classification

A drop down field indicating the type of validation check. Values include"\:

  • Source data validation - These validations perform checks on the data used as inputs to a process.
  • Valuation Data Validation - These validation checks are performed on the output values of a process. For example if the process calculates a NAV price, any checks performed on that NAV price would fall under this classification.
  • Value Add Function - A validation check performed on the values within analysis reporting produced at the end of a process

Business Description

A short description of the check being performed by this validation.

Validation Formula (will fail if)

The formula used by the check calculation.

Note: This formula is only displayed for information purposes. pControl rules contains the technical formula used by this check.
Validation Calculation formula

Each pControl validation is specified as an equation with a left hand side (LHS) and right hand side (RHS); this equation forms the 'failure condition' for the validation.

Where the failure condition is specified as:

$\small u100.base\_ mv\_ close < > u100.base \_ capital \_ value\ +\ u100.base \_ accrued\_ interest$

LHS of the equation: $\small u100.base\_mv\_close$

RHS of the equation: $\small u100.base\_capital\_value\ + u100.base\_accrued\_interest$

The validation logic is as follows:

If LHS = RHS, then

  • Check Value = 0 (validation always passes unless tolerance is set to 0)

Else, if LHS = 0, then

  • Check Value = a value other than 0.

Else

  • $\small Check\ value\ =\ \left (\frac{RHS}{LHS} \right )\ -1$

The 'check value' is then compared against the tolerances set for the validation (if the validation is not simply a true/false validation). Where the check value is greater than the tolerance specified, an exception will be raised. Check values, (and exceptions) are reported in the following places:

  • Process status messages
  • Process detail worktables, (e.g. A07, A02)
  • Exception events (if the tolerance is breached)
Note

The business description and validation formula displayed on validation entities is for information purposes only. pControl rules contain the actual validation formula used by this check.

 

 Validation Configuration

Attribute

Description

Validation Configuration

Rules File

This field contains the name of the rules file which is executed when this validation check is performed.

Note: The contents of this rule file cannot be viewed via the web or processing modes.

Exception Event Calculation Details Rules

A drop down list of the criteria fields an auto clearance rule requires, and to support any APIs needed.

Note: Auto clearance of exceptions will only occur on those fund entities configured to allow auto clearance. For further details, refer to Auto Clearance .

Run On Initialisation

If selected, this validation check will be run when the process is run in initialisation mode.

Validation Type

This identifies what type of data this validation will be performed on, i.e., Holding, Transaction, Product, Category or Entity.

Comparison Type

This determines how the value calculated by the validation will be compared to the tolerance that has been set for the validation for this entity:

Absolute: if the absolute calculated value is greater than the tolerance value then tolerance is breached.

Less than: if the calculated value is less than tolerance value then the tolerance is breached.

Greater than: if the calculated value is greater than the tolerance value then the tolerance is breached.

Not equal: if the calculated value is not equal to the tolerance value then the tolerance is breached.

Equal: if the calculated value is equal to the tolerance value then the tolerance is breached.

Not applicable: if the validation does not require a comparison to the tolerance, e.g. it is testing for a zero cost base or zero market value so is not a comparison test but a TRUE/FALSE test.

Tolerance Value Type

This determines whether the value calculated by the validation and the tolerance specified is a:

  • percentage figure
  • basis point figure or
  • flat amount.

Note: The validation entities will be pre-configured to have the correct value type and should only be updated after careful consideration of the impact.

Format String

This determines how the output from the validation will be presented via the status messages and in pControl worktables, e.g. the format mask determining the number of decimal places shown, whether the value will be shown as a percentage, commas for thousand separators and any other standard formatting masks supported by the pControl 'FormatDecimal' function.

Examples of formatting masks that could be entered into this field are provided below:

  • 0.0000 - show value with 4 decimal places.
  • 0.00% - present the value as a percent, showing two decimal places and with a percent sign.
  • ###,###.00 - present the value showing two decimal places with commas for thousand separators.
  • AUD0.00 - present the value showing two decimal places with the text 'AUD' prefixing the number.
  • 0.00bp - present the value showing two decimal places with the text 'bp' suffixing the number.

Status / Override Message

Text entered into this field will be used to prefix all status messages and exception events produced from this validation, e.g. if the text string 'SECURITY TEAM' was entered, then any status message or exception event produced by this validation would have the text string 'SECURITY TEAM' at the start of the message.

The intention of this function is so that specific validations can easily be identified for specific purposes, e.g. to allocate to a specific team of people for resolution.

Fixed Comment Lists

Multiple pre-configured comment lists that can be linked to a validation entity according to various reporting locations. If configured, users are able to utilise the fixed comment list when overriding validation exceptions of appending comments in control records.

When adding the fixed comment lists, the following fields should be populated:

  • Location - the location where a particular fixed comment list is assigned to.
  • Fixed Comment Lists - drop-down lists of pre-defined comments that have been configured in system administration mode.
  • Mandatory - if selected, the fixed comment field in exception events / append comment will be populated with the comment from the selected fixed comment list.

Tolerances

Check Against Tolerances

Determines whether the validation will actually check the calculated value against a tolerance or merely calculate the value and report it through status messages and output to the pControl worktables.

Typically, this box will always be selected. If this was not selected, no exceptions would ever be raised for the validation check, however, the results would still be produced and reported.

Use Tolerance Entity rather than L200 Tolerances?

Some validations allow the use of a tolerance entity rather than the tolerance as specified in the L200: Tolerance worktable. This must be selected if the tolerance entity is to be used.

Validation Detail Output

Create Summary Record

If selected, an additional record will be written to the relevant detail validation worktable showing the total count of process runs, as well as the total count of validation checks performed and tolerance level incurred.

E.g. "Total runs = 26 | OK = 23 | Exception = 3"

Always Export All Detail Output

If selected, a record will be written to the relevant detail validation worktable showing the results of all validation checks performed. This includes validation checks that passed, as well as those that breached their tolerance.

The setting of this attribute at validation entity level means all entities which perform this validation check will have a record written to the detail validation worktable, regardless of whether the "Export ALL detail output" attribute at entity level is selected or not.

Override Summary

Group for Override Summary

Free format text field. The value in this field is used for classifying exceptions on the Exceptions View.

Sort Order within the Group

This determines the order that the validation entity codes appear within the exception event summary groups. Number '0' is the highest sort order.

Key Indicator

 

Key Indicator Category Code

Contains a drop down list of Key Indicator category codes. If selected, the results of this validation check, will be compared to the tolerance, and reported on the VMAS report, and Fund Summary screen.

 

 Validation entity 'applies to' data attributes

The 'applies to' attributes on a validation entity are used to determine which types of instrument or investment category, a specific validation will be applied to. Certain validations are only applicable for specific instrument types i.e. various accrued income checks are only applicable to, or sensible to perform on fixed income securities.

The specific instrument types or categories that are validated by a specific validation can be controlled using the criteria and instrument groups/types attributes.

Attribute

Description

Criteria

Determines whether the selected instrument types (those present on the right hand side of the selection box) are to be included in this validation or ‘excluded by this validation.

  • If the criteria is set to ‘Include’ and there are no instrument types selected, the validation will not run.
  • If the criteria is set to ‘Exclude’ and there are no instrument types selected, the validation will process all instrument groups / types present in the import worktables.

Instrument Groups / Types

This is the attribute that can be used to select the set of instrument types which will either be included in this validation or excluded from it. The list of instrument groups / types is driven by the instrument groups and types present in the ‘L220: Instrument Type Mappings’ static worktable where the mapping from client specific instrument codes to pControl instrument types is maintained.

For example, when loading investment holdings via ‘U100: Investment Holdings’, data enrichment validation will map client specific instrument codes into corresponding pControl instrument types maintained in ‘L220: Investment Type Mappings’.

Categories

Where categories are shown in the 'selected' list, (right hand side), this validation check will only be performed on data records with the category code listed.

If the value of any category exceeds the tolerance set for this validation check, the exception will be reported in the NAV Movement Analysis report, as well as all the other places exceptions are reported.

Example

In processes where [CAT002] - UP Category Impact check is turned on, pControl will perform this check on all investment holding records with a category code that matches one of the codes from the list in the right hand column.

 

 Validation entity 'Override Events' attribute

The 'Override Events' attribute on the validation entity provides the ability to order exceptions, by the priority of the validation as well as by location. The sort order assigned to the validation, i.e. 1, 2, 3 is defined on the validation entity and different orders can be set for different locations. To sort by location, the location specified on the validation entity must match the Price Release Location selected on the fund, as shown below.

The sort order number may or may not appear on the Exceptions view, subject to data view that is applied.
